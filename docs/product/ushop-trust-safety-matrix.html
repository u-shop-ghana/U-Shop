01 — Threat Model & Mitigation

Platform Threat Model

The five threats below were selected because they are specific to U-Shop's architecture — the combination of user-generated policy text, Paystack's collect-and-transfer escrow, high-value tech inventory, and a student user base creates a distinct attack surface. Generic OWASP checklists do not fully address this surface.

Security Philosophy

The biggest risk to U-Shop is not a technical breach — it is a trust failure that goes viral on campus. One widely-shared story of a GH₵ 2,000 laptop scam that the platform failed to resolve will cause months of growth stagnation. These mitigations are therefore brand protection as much as they are technical controls.

Threat 1 — XSS via Custom Policy & Store Content

CRITICAL

Cross-Site Scripting (XSS) via User-Generated Policy Text

A malicious seller injects JavaScript into their custom store name, bio, return policy text, or warranty notes. When other users load the store page, the script executes in their browser session — potentially stealing session tokens, redirecting buyers to phishing payment pages, or silently modifying checkout totals.

⚠ Attack Surface
  • !Store bio field — rendered on the public store page, SSR'd into HTML for SEO. A single unescaped <script> tag here executes for every visitor.
  • !Custom return policy text — free-text up to 500 chars. Rendered in the listing sidebar and checkout summary. High-impact target.
  • !Product description — supports markdown-like formatting, widening the injection surface significantly if not sanitised.
  • !Store name / handle — rendered in OG images, email notifications, and admin dashboard. An XSS payload in an email's HTML is a separate attack vector.
  • !Dispute evidence descriptions — rendered in the admin panel. Admin panels are high-value XSS targets because admins have elevated session privileges.
✓ Mitigations (Layered)
  • ✓Layer 1 — Input Sanitisation (Express): All free-text inputs are sanitised server-side using isomorphic-dompurify before being written to the database. Strip all HTML tags from store name, bio, and handle. For policy text and descriptions: strip all tags except a defined allowlist (b, i, br, ul, li). No script, style, a, img, iframe — ever.
  • ✓Layer 2 — Output Escaping (React/Next.js): React escapes all string values rendered via JSX by default. Never use dangerouslySetInnerHTML with user-generated content. If markdown rendering is needed, use a sanitising renderer (e.g. react-markdown with the rehype-sanitize plugin).
  • ✓Layer 3 — Content Security Policy (CSP): Set a strict CSP header on all responses. script-src 'self' means even if a script tag is somehow rendered, the browser refuses to execute it unless it originates from your own domain.
  • ✓Layer 4 — Zod Schema Validation: All API inputs validated with Zod before sanitisation runs. Fields with maximum length constraints (500 chars for policy text) are enforced here first.
Implementation — Express Sanitisation Middleware
// apps/api/src/middleware/sanitize.ts
// Run this on ALL user-generated text before it touches the database.
// DOMPurify strips malicious content; Zod has already enforced length limits.

import DOMPurify from 'isomorphic-dompurify';

// No HTML allowed in these fields — strip everything
export function sanitizePlainText(input: string): string {
  return DOMPurify.sanitize(input, { ALLOWED_TAGS: [], ALLOWED_ATTR: [] })
    .trim()
    .replace(/\s+/g, ' '); // Collapse multiple spaces
}

// Limited formatting allowed in policy text and descriptions
export function sanitizeRichText(input: string): string {
  return DOMPurify.sanitize(input, {
    ALLOWED_TAGS: ['b', 'i', 'em', 'strong', 'br', 'ul', 'ol', 'li', 'p'],
    ALLOWED_ATTR: [],  // No attributes — blocks onclick, style, href, etc.
    FORBID_TAGS: ['script', 'style', 'iframe', 'form', 'input'],
    FORBID_ATTR: ['onerror', 'onload', 'onclick', 'style'],
  });
}

Threat 2 — Fake Listings & "Brick in a Box" Fraud

CRITICAL

Fraudulent Product Listings — Counterfeit, Stolen, and Misrepresented Devices

A seller lists a MacBook Pro at below-market price. The buyer pays via MoMo. The seller ships a box of bricks, a non-functional device, or a device that is stolen and IMEI-blocked. Alternatively, the seller lists as "Excellent" condition what is functionally broken. These are the highest-frequency fraud scenarios on any second-hand tech marketplace.

⚠ Attack Variants
  • !Brick-in-Box: Buyer pays, receives a box containing bricks or junk. Seller claims delivery was made. Escrow auto-releases after 7 days if buyer doesn't dispute in time.
  • !Condition Fraud: Device listed as "Used — Good" arrives in "For Parts" condition. Battery health screenshot is faked using a photo editor.
  • !Stolen Device: Seller lists a laptop or phone with a fraudulent or blocked IMEI/serial number. Buyer later discovers the device is blacklisted and non-functional on networks.
  • !Counterfeit: Replica iPhone or laptop sold as genuine. Particularly high risk at Tier 2/3 price points where deals "seem too good".
  • !Account Takeover Listing: Fraudster gains access to a high-rated seller account and posts fraudulent listings using its reputation to avoid suspicion.
✓ Mitigations (Layered)
  • ✓Mandatory IMEI/Serial Disclosure: All phone listings require IMEI input. All laptop listings above GH₵ 500 require serial number. These are stored on the listing record and displayed to buyers before purchase.
  • ✓Illustrated Condition Grade Guide: Each of the 6 condition tiers is defined with photos, specific criteria (battery %, scratch tolerance, port functionality). Sellers must select a grade that triggers required evidence — e.g., "Used — Good" mandates a battery health screenshot. Per PRD: this single measure is estimated to halve dispute volume.
  • ✓Mandatory Photo Requirements per Condition: "Used — Good" and below require photos of all four corners, ports, and the screen. These are uploaded before the listing can be activated.
  • ✓Escrow Protects the Buyer: Funds are never released until buyer confirms delivery. The 48-hour dispute window (opening from confirmed delivery or from the moment the buyer clicks Received) is the buyer's final safety net.
  • ✓Seller Reputation System: Verified student sellers, established review ratings, and transaction history are surfaced prominently. New sellers with zero reviews are visually flagged.
  • ✓Report Stolen Device Button: Visible on every listing. Links to a Ghana Police Service IMEI check integration (Phase 2). Any reported listing triggers an immediate escrow freeze on active orders.
Implementation — Listing Activation Guard (Express)
// apps/api/src/services/listing.service.ts
// Called before a listing can transition from DRAFT → ACTIVE.
// A listing that does not meet these requirements simply cannot go live.

async function validateListingForActivation(listing: Listing): Promise<void> {
  const errors: string[] = [];

  // Rule 1: At least 2 photos required for any used item
  if (listing.condition !== 'NEW' && listing.imagePaths.length < 2) {
    errors.push('Used items require at least 2 photos');
  }

  // Rule 2: Battery screenshot required for phones/laptops in Used conditions
  const BATTERY_REQUIRED_CONDITIONS = ['USED_GOOD', 'USED_FAIR', 'USED_EXCELLENT'];
  if (BATTERY_REQUIRED_CONDITIONS.includes(listing.condition) &&
      listing.category.slug === 'phones' || listing.category.slug === 'laptops') {
    if (!listing.batteryScreenshotPath) {
      errors.push('Battery health screenshot required for this condition grade');
    }
  }

  // Rule 3: IMEI required for all phones
  if (listing.category.slug === 'phones' && !listing.imei) {
    errors.push('IMEI number is mandatory for all phone listings');
  }

  // Rule 4: Serial number required for high-value laptops
  if (listing.category.slug === 'laptops' &&
      listing.price >= 500 && !listing.serialNumber) {
    errors.push('Serial number required for laptop listings above GH₵ 500');
  }

  if (errors.length > 0) {
    throw new ValidationError(errors.join(' | '));
  }
}

Threat 3 — Paystack Webhook Manipulation & Replay Attacks

CRITICAL

Forged or Replayed Paystack Webhooks

An attacker discovers your webhook URL (it is often predictable) and sends a forged charge.success event to your Express server. If your server doesn't verify the HMAC signature, it will process the fake payment, mark the order as paid, create an escrow record — and the attacker gets free goods. A replay attack sends a genuine past webhook a second time to trigger a double-escrow-release.

⚠ Attack Variants
  • !Webhook Forgery: Attacker crafts a fake charge.success payload and POSTs it to /webhooks/paystack. Without HMAC verification, the server accepts it as legitimate payment.
  • !Replay Attack: Attacker captures a real webhook (from a sniffed connection or by placing a real order) and replays it. Without idempotency checks, this triggers a second escrow creation or second release.
  • !Redirect Exploitation: Attacker manually visits the callback URL (/checkout/complete?reference=abc123) without paying. If the server trusts the redirect alone, the order is marked paid.
  • !Race Between Redirect and Webhook: The payment page arrives before the webhook is processed. A naive implementation might trust the redirect and mark the order paid before verification.
✓ Mitigations (Implemented)
  • ✓HMAC-SHA512 Signature Verification (Layer 1): Every webhook is verified by computing HMAC-SHA512(rawBody, PAYSTACK_SECRET_KEY) and comparing it to x-paystack-signature using crypto.timingSafeEqual(). Failure → 401, logged immediately. This is the primary authentication mechanism. The webhook URL itself is not a secret.
  • ✓Idempotency via WebhookEvent Table (Layer 2): Every event is stored with Paystack's unique event ID as externalId (unique constraint in DB). Before processing any event: check if it already exists. If processed = true, return 200 immediately without side effects. This stops replay attacks cold.
  • ✓Raw Body Parsing (Layer 3): The webhook route is registered before express.json() and uses express.raw() to preserve the raw buffer. HMAC is computed on the exact bytes Paystack sent — not a re-serialized JSON object. This is the most common implementation mistake and is specifically guarded against.
  • ✓Redirect Never Trusted (Layer 4): The checkout callback page (/checkout/complete) calls the Express API to fetch the order's current status. It never marks orders as paid directly. The webhook is the source of truth. If the webhook hasn't fired yet, the page polls every 2 seconds for up to 30 seconds.
Timing-Safe Signature Comparison (the correct way)
// crypto.timingSafeEqual prevents timing attacks.
// A naive comparison (hash === signature) leaks information about
// how many characters matched via the time it takes to return false.
// timingSafeEqual always takes the same time regardless of match position.

const signature = req.headers['x-paystack-signature'] as string;

if (!signature) {
  return res.status(401).send('Missing signature');
}

const expectedHash = crypto
  .createHmac('sha512', process.env.PAYSTACK_SECRET_KEY!)
  .update(req.body) // req.body = raw Buffer — NOT JSON.parse()'d
  .digest('hex');

// Both buffers must be the same length before comparison
const sigBuffer  = Buffer.from(signature, 'hex');
const hashBuffer = Buffer.from(expectedHash, 'hex');

if (sigBuffer.length !== hashBuffer.length ||
    !crypto.timingSafeEqual(sigBuffer, hashBuffer)) {
  console.warn('⚠ Paystack webhook signature FAILED — potential forgery attempt');
  return res.status(401).send('Invalid signature');
}

// Only parse the body AFTER verification passes
const event = JSON.parse(req.body.toString()) as PaystackEvent;

Threat 4 — Inventory Race Conditions at Checkout

HIGH

Concurrent Checkout — Overselling the Same Item to Multiple Buyers

Two buyers both see "1 unit in stock" and both click "Buy Now" within milliseconds of each other. Both API requests pass the stock-check validation simultaneously. Both orders are created and sent to Paystack. Both buyers pay. One item. Two confirmed orders. The seller cannot fulfil both; one buyer will be defrauded by the platform's own checkout flow.

⚠ Why This Happens
  • !Two requests hit Express simultaneously. Both call listing.findUnique() and read stock = 1. Both pass the stock validation (if stock < 1, throw error).
  • !Both proceed to create an Order and call Paystack to initialise a transaction. Both buyers get sent to the payment page.
  • !Both buyers complete payment. Both charge.success webhooks arrive. Both orders are marked PAYMENT_RECEIVED. Stock is now at -1.
  • !The same race condition can occur at the escrow release stage: a buyer double-clicks "Confirm Receipt", firing two simultaneous release requests.
✓ Mitigation — Atomic Database Operations
  • ✓Atomic Stock Decrement with Conditional WHERE: Instead of read-then-update (the race condition), use a single atomic SQL statement: UPDATE listings SET stock = stock - 1 WHERE id = $1 AND stock >= 1. PostgreSQL executes this atomically. If two concurrent requests reach the DB, one will find stock = 1 and decrement successfully; the second will find stock = 0 and update 0 rows — detectable as a failure.
  • ✓Optimistic Locking for Escrow Release: The escrow release query includes a WHERE status = 'HOLDING' condition. The first request updates the escrow to RELEASING; the second finds no rows in HOLDING state and fails gracefully with a 409 Conflict.
  • ✓Prisma Transaction Wrapping: All stock decrement + order creation logic runs inside a single prisma.$transaction(). If any part fails, the entire operation rolls back — preventing partial states (order created, stock not decremented).
  • ✓Frontend Debounce: The "Buy Now" button is disabled immediately on click and shows a loading state. This stops accidental double-clicks at the UI layer before they hit the API.
Atomic Stock Reservation — The Correct Pattern
// The WRONG way (read then write — race condition):
// const listing = await prisma.listing.findUnique({ where: { id } })
// if (listing.stock < quantity) throw new Error('Out of stock')    ← TOCTOU bug
// await prisma.listing.update({ data: { stock: { decrement: quantity } } })

// The RIGHT way — single atomic conditional UPDATE:
// Prisma $executeRaw returns the number of rows affected.

const rowsAffected = await prisma.$executeRaw`
  UPDATE listings
  SET    stock = stock - ${quantity}
  WHERE  id    = ${listingId}
  AND    stock >= ${quantity}
  AND    status = 'ACTIVE'
`;

// 0 rows affected = stock was already 0 when we tried to decrement.
// The operation failed safely. No partial state exists.
if (rowsAffected === 0) {
  throw new ConflictError('This item just sold out. Please choose another listing.');
}

// After stock is reserved, THEN create the Order and call Paystack.
// All inside prisma.$transaction() for rollback safety.

Threat 5 — Stolen Device Laundering

HIGH

Stolen or Blacklisted Tech Devices Listed for Sale

A thief steals a laptop or phone, lists it on U-Shop at an attractive price, receives payment via escrow, and disappears. The buyer later discovers the device's IMEI is blacklisted by the network (for phones) or that the laptop's serial number is registered to someone else via manufacturer theft databases. U-Shop faces legal liability and severe reputational damage.

⚠ Platform Risk
  • !Ghana Police Service can hold a platform liable for knowingly facilitating the sale of stolen goods if no verification process exists.
  • !A buyer who purchases a blacklisted phone (SIM card rejected by all networks) has received a worthless item. Platform escrow has already released funds.
  • !University campus context amplifies reputational risk — students talk. One stolen laptop story circulates immediately via WhatsApp groups and destroys trust rapidly.
  • !Repeat offenders: fraudsters will create multiple accounts after being banned unless identity verification ties accounts to real identities.
✓ Mitigations
  • ✓IMEI Mandatory for All Phone Listings: Stored at listing creation. Displayed prominently to buyers with a "Check IMEI" button that links to a free IMEI checker tool. In Phase 2: integrate with a paid IMEI blacklist API for automatic verification.
  • ✓Serial Number Required for High-Value Laptops: All laptops above GH₵ 500 require the serial number. Admin can cross-reference against manufacturer theft databases manually or via API (Phase 2).
  • ✓"Report Stolen Device" Button: Visible on every active listing. Opens a form linked to the buyer's account. Any report triggers an automatic escrow freeze on active orders for that listing, pending admin review within 24 hours.
  • ✓Account Suspension on Confirmed Fraud: Any seller account confirmed to have listed a stolen device is permanently banned. Their Paystack payout is blocked pending police coordination. Their verified identity (student ID or Ghana Card for non-students) is flagged in our system.
  • ✓Seller Identity Requirements: All sellers must verify with a Ghana Card or student ID before activating a store. This creates a real-world identity link that deters casual thieves.
02 — Escrow & Payment Lifecycle

Escrow Lifecycle — Operational Rules

The escrow system is not a feature — it is the foundational trust mechanism of U-Shop. Every operational rule below was derived from the PRD's blind spots section and the escrow technical specification. These rules are non-negotiable and must be implemented in code exactly as written here.

The Core Principle

Escrow in U-Shop is virtual. Money flows: buyer pays → 100% lands in U-Shop's Paystack account balance → our database ledger tracks what belongs to whom → on release, we call Paystack Transfer API to send the seller's cut to their MoMo. The database is the escrow system. Paystack is the payment rails.

The 7-Day Countdown — Notification Timeline

The auto-release timer is a financial safety feature, not just UX. A buyer who misses the dispute window because they were never notified is a platform liability. The notification schedule below is therefore mandatory — not optional — to implement before launch.

0

Payment Confirmed — Escrow Created

Paystack fires charge.success webhook. Express verifies HMAC, creates Escrow record (HELD), sets escrow_release_at = paid_at + 7 days. Seller notified immediately: "You have a new order. Arrange delivery within 48 hours." Buyer notified: "Payment received. Your funds are secure in escrow."

Day 0 — T=0BullMQ jobs scheduled for Day 3, 5, 6, and 6.5 reminders
→

Seller Arranges Delivery / Meetup

Seller updates order status to IN_TRANSIT or arranges a campus meetup via the in-app message thread. The message thread is the permanent record of all delivery commitments. Sellers are advised: any delivery promise made in this thread can be used as evidence in a dispute.

Day 0–2No notifications sent. Platform assumes delivery is in progress.
3

Day 3 — First Reminder to Buyer

Email + in-app notification to buyer: "Your order from [Store Name] should have arrived or be arriving soon. Once you receive and inspect the item, please confirm delivery. If there is a problem, open a dispute now — don't wait."

Day 3 — T+72hTone: friendly, informational. No urgency language yet.
⚠ If the buyer still hasn't received the item, this is the correct moment to open a dispute. The Day 3 reminder should prominently feature the "Open Dispute" button alongside "Confirm Delivery".
5

Day 5 — Second Reminder (Urgency Introduced)

Email + in-app + push notification: "Have you received your item? Payment to the seller will be automatically released in 48 hours if you do not confirm delivery. If there is ANY problem with your order, open a dispute before then."

Day 5 — T+120hTone: urgent. Two clear buttons: Confirm Delivery and Open Dispute — equal prominence.
6

Day 6 — Final Warning (24 Hours Remaining)

Email + in-app + push notification: "🔴 FINAL NOTICE: Payment to the seller releases automatically in 24 hours. If you have NOT received the item, or if the item is damaged, broken, or not as described — open a dispute NOW. This is your last opportunity to protect your payment."

Day 6 — T+144hTone: maximum urgency. Dispute button is the primary CTA at this stage.
6½

Day 6.5 — Last Chance (6 Hours Remaining)

Final notification, 6 hours before auto-release: "⚠ Payment releases in 6 hours. If you have any concerns about your order, this is your absolute last chance to open a dispute. After release, platform intervention requires a post-release dispute which has more limited remedies."

Day 6.5 — T+156hSMS sent if phone number is on file, in addition to email and in-app.
7

Day 7 — Auto-Release Fires

The hourly cron job (escrow-release.job.ts) detects the order: status = PAYMENT_RECEIVED and escrow_release_at <= NOW(). It calls EscrowService.releaseToSeller(orderId, 'AUTO_7DAY') — the same code path as a buyer-triggered release. Seller's wallet is credited. Seller receives an email: "Great news — your payment has been released. GH₵ X is now available in your wallet."

Auto-Release — T+168hrelease_trigger = 'AUTO_7DAY' stored on EscrowLedger for audit trail.
⚠ Auto-release does NOT close the dispute window. Buyers may still open a post-release dispute (with more limited remedies) within 24 hours of the auto-release if they can demonstrate the notification schedule was not delivered to them.

Operational Rules — The Escrow Rulebook

These 14 rules govern every decision the escrow system makes. They are written as implementation contracts — if your code does not satisfy all of them, it is not production-ready.

#RuleRationaleEnforced By
R-01 Escrow is created only after a verified charge.success webhook — never on redirect callback alone The redirect can be manually triggered by anyone without paying. The webhook is cryptographically signed proof of payment. Webhook HMAC verification + business logic gate
R-02 The escrow auto-release timer is set to exactly 7 days from paid_at, not from order creation paid_at is when the buyer's funds were confirmed. Creating the order might precede payment by minutes. Using order creation would shortchange the buyer. escrow_release_at = paidAt + 7 days in handleChargeSuccess
R-03 Escrow can only be released if its current status is HELD Prevents double-release. Two concurrent release calls cannot both succeed — the second finds no row in HELD state. Atomic DB update: WHERE status = 'HELD'
R-04 Only the buyer of the specific order can manually trigger escrow release Prevents sellers from releasing their own escrow. Prevents one buyer from releasing another buyer's order. Ownership check: order.buyerId === req.user.id
R-05 Opening a dispute immediately freezes escrow — auto-release is suspended A disputed order must not auto-release while evidence is being reviewed. The buyer's protection window must not expire during arbitration. order.status = 'DISPUTED' prevents escrow cron from processing it
R-06 A buyer can open a dispute while the order is in PAYMENT_RECEIVED, PROCESSING, or IN_TRANSIT status Buyers must not have to wait for the seller to update the order status before they can protect themselves. Dispute creation validates against the allowed status set
R-07 After a buyer manually confirms delivery, a 48-hour dispute window remains open The buyer may have confirmed delivery before fully inspecting the device (e.g., confirmed at handoff, discovered damage later). 48 hours is the minimum fair inspection period. dispute_window_closes_at = deliveredAt + 48h; dispute endpoint checks this
R-08 Wallet crediting is atomic with escrow status update — both succeed or both roll back Money can never be "released" without being credited to the wallet. A failed wallet credit with a successful escrow update would strand funds. Prisma $transaction() wrapping both operations
R-09 If a Paystack transfer fails, the wallet debit is reversed immediately The seller's wallet is debited when the payout is initiated. A failed transfer must refund the wallet balance — the seller should not lose access to their money. transfer.failed webhook handler re-credits the wallet
R-10 The release_trigger field on EscrowLedger is always populated Audit trail: why was this escrow released? Values: BUYER_CONFIRMED, AUTO_7DAY, ADMIN_RELEASE, ADMIN_REFUND. Required for dispute investigation and financial reconciliation. All release code paths set this field explicitly
R-11 A seller cannot withdraw wallet funds while a related dispute is OPEN or UNDER_REVIEW Prevents the seller from withdrawing disputed funds before the dispute is resolved, making refunds impossible. Payout request endpoint checks for open disputes related to the wallet's orders
R-12 Minimum payout amount is GH₵ 20 Per PRD specification. Prevents microwithdrawal spam that generates excessive Paystack transfer fees. Payout Zod schema: amount: z.number().min(20)
R-13 The platform fee is never returned to the seller, even on full refunds Refunds credit the buyer from the held_amount (the seller's cut). The platform_fee portion is U-Shop's earned revenue and is retained. Sellers are informed of this in the Terms of Service. Refund amount = escrow.held_amount, not order.total_amount
R-14 All webhook events are stored in webhook_events before any processing If processing fails mid-way, the raw event is preserved. A reconciliation cron can replay unprocessed events. No payment event is ever permanently lost. Webhook handler: persist event first, respond 200, then process
03 — Dispute Resolution Playbook

Dispute Resolution — Operational Playbook

Disputes are the moment of truth for platform trust. A dispute resolved quickly, fairly, and transparently turns a negative experience into a story about a platform that works. A dispute handled poorly becomes the WhatsApp story that kills the platform.

The Policy Hierarchy — What Overrides What

U-Shop has a three-tier policy hierarchy. The most important rule: higher tiers always supersede lower tiers, without exception. A seller's "No Returns" policy cannot override the platform's minimum buyer protections. This hierarchy must be disclosed to sellers at store creation and agreed to in the Seller Terms of Service.

1

Tier 1 — Platform Minimum Buyer Protections (Cannot be waived)

These apply to every transaction regardless of seller policy. They are the floor — sellers cannot go below these protections in their custom policies.

  • Funds are always held in escrow for a minimum of 24 hours after confirmed delivery before any release
  • Buyers can always open a fraud or DOA (Dead on Arrival) dispute within 48 hours of delivery confirmation, regardless of seller's stated return window
  • Confirmed counterfeit, materially misrepresented, or stolen items always qualify for a full refund of the held_amount, irrespective of any seller policy
  • Items that are not as described in a material way (e.g., listed as "Functional" but device will not power on) qualify for dispute, regardless of seller's "No Returns" policy
  • A seller's "No Returns" policy can never be used to deny a buyer the right to open a dispute — it can only limit the grounds for accepting a return for change-of-mind
2

Tier 2 — Seller Custom Policy (Within platform limits)

Sellers configure their return window, warranty terms, and refund method through the structured policy builder. These govern change-of-mind returns and general warranty claims — but are always subject to Tier 1 overrides.

  • Return window settings (No Returns / 24H / 3D / 7D / 14D / 30D) govern change-of-mind returns only
  • Warranty coverage settings apply to hardware defects beyond the DOA window
  • Policy is immutably snapshotted at checkout — a seller changing their policy post-sale has zero effect on existing orders
  • Platform ToS explicitly states: seller policy is binding on the seller, but Tier 1 protections supersede it whenever there is conflict
3

Tier 3 — Buyer-Seller Negotiated Resolution

Before admin intervention, parties are encouraged to resolve disputes directly through the in-app order message thread. A buyer-seller agreement (e.g., partial refund, exchange) can be executed by admin without formal arbitration. This preserves both parties' reputation and reduces admin workload.

  • Agreement must be stated explicitly in the order message thread
  • Admin implements the agreed resolution (credit the wallet, issue partial refund)
  • Both parties confirm resolution — dispute is marked CLOSED

The Standard 8-Step Dispute Protocol

Every dispute — regardless of category — follows this protocol. SLAs are measured from the moment the dispute is created. Missing SLAs must be escalated to the next admin tier.

1

Buyer Opens Dispute — Escrow Frozen

Buyer selects a reason (ITEM_NOT_RECEIVED, NOT_AS_DESCRIBED, DAMAGED_IN_TRANSIT, WRONG_ITEM_SENT, DOA, COUNTERFEIT) and submits a written description (min 20 chars) with optional photo/video evidence (up to 5 files). Express atomically sets order.status = 'DISPUTED' and creates a Dispute record. Escrow auto-release is immediately suspended. Both buyer and seller are notified. Buyer is told: "Your dispute has been received. Payment to the seller is frozen pending review."

⏱ T+0 — Immediate, automated
2

Seller Notified — Response Window Opens

Seller receives email + in-app alert: "A buyer has opened a dispute on Order #[ID]. Please respond with your evidence within 24 hours. Failure to respond will result in the dispute being decided in the buyer's favour." The seller can upload counter-evidence (photos of item before dispatch, proof of delivery, tracking confirmation) via the admin-facing dispute panel.

⏱ Seller response deadline: T+24h
3

Admin Acknowledges & Reviews Evidence

An admin picks up the dispute from the queue. Status updated to UNDER_REVIEW. Admin reviews: (a) the order's message thread, (b) buyer's submitted photos/video, (c) seller's counter-evidence, (d) the listing photos at time of sale, (e) the immutable policy snapshot from the order record. Admin sends an acknowledgement to both parties: "We are reviewing your dispute. We aim to reach a decision within 72 hours."

⏱ First admin contact: T+24h from dispute creation
4

Admin May Request Additional Evidence

If evidence is insufficient, admin can request: additional photos from either party, a video unboxing (if the buyer has the item), or a third-party inspection report (for high-value disputes above GH₵ 1,000). Each request resets the SLA clock for that party by 24 hours. Maximum of one additional evidence request per party before forcing a decision.

⏱ Evidence response window: 24h per request
5

Admin Consults the Policy Hierarchy

Before ruling, admin explicitly documents which tier of the hierarchy governs the decision. The internal admin note must state: which tier applies, why the seller's custom policy does or does not apply, and the specific evidence that supports the ruling. This documentation is the legal shield if either party escalates to Ghana's Data Protection Commission or court.

⏱ Decision deadline: T+72h from dispute creation (T+48h from admin acknowledgement)
6

Admin Issues Ruling

Ruling in Buyer's Favour: Escrow status → REFUNDED. Platform issues a refund of escrow.held_amount to the buyer's original payment method (card) or U-Shop wallet credit (MoMo). The seller receives nothing. Platform retains the platform_fee. Seller's reputation score is penalised (affects their listing visibility ranking).

Ruling in Seller's Favour: Escrow status → RELEASED. Funds credited to seller's wallet. Buyer's dispute is marked RESOLVED_SELLER. If the buyer filed a false dispute (multiple instances), account is flagged for review.

⏱ Ruling issued: T+72h maximum from dispute open
7

Executing the Resolution

Admin executes the ruling via the admin panel: POST /api/admin/disputes/:id/resolve with { outcome: 'BUYER' | 'SELLER', adminNotes: '...', partialRefundAmount?: number }. The system atomically updates all affected records: Dispute, EscrowLedger, Order, and (if applicable) WalletTransaction. Both parties are notified with the resolution reason and next steps.

⏱ Execution: Immediate on admin action
8

Post-Resolution Actions & Account Management

After resolution: (a) The losing party has 7 days to raise a formal appeal via email to the Trust & Safety team — this is a manual, human-only process. (b) If seller is found to have committed intentional fraud (counterfeit listing, brick-in-box), their account is permanently suspended, their remaining wallet balance is frozen pending police coordination, and the listing is removed with a public note. (c) If the buyer is found to have filed a false DOA claim (item was functional), this is recorded. A third false claim results in account suspension.

⏱ Appeal window: 7 days from ruling notification

The Core Conflict — Case Study: Broken Laptop + "No Returns / As Is" Policy

This is the scenario that will happen within weeks of launch. Here is the exact ruling protocol.

⚖️

Scenario: Buyer receives a non-functional laptop. Seller's policy: "No Returns — Sold As Is"

Order value: GH₵ 1,800 · Condition listed: "Used — Good" · Listing stated: "All ports functional, battery above 80%"

Fact-Finding

Admin pulls the order's immutable policy_snapshot from the database. Confirms: returnWindow: "NO_RETURNS", warrantyCoverage: "DOA_ONLY". Admin also pulls the listing's images at time of sale (stored in Supabase Storage under the listings/{listingId}/ path — these are never deleted from completed orders). Admin reviews the order message thread for any pre-sale representations by the seller.

Key Question: Does "No Returns / As Is" apply here?

"No Returns / As Is" is a valid Tier 2 seller policy that governs change-of-mind returns. A buyer who simply decided they don't want the laptop anymore has no claim under a "No Returns" policy. However, "Not as described" is not a return — it is fraud protection, which falls under Tier 1. If the listing stated "All ports functional, battery above 80%" and the device arrived non-functional, the seller's listing was materially misrepresented, regardless of their return policy.

The Critical Distinction

Admin must determine: Is this a change-of-mind return (buyer gets buyer's remorse → "No Returns" policy applies → seller wins) or a material misrepresentation (item is not what was sold → Tier 1 platform protection applies → buyer wins)? This distinction is determined by: (a) the exact wording of the listing description vs. the condition of the received item, and (b) the photographic evidence from both parties.

Evidence Standard for DOA / Not As Described

To win a DOA / material misrepresentation dispute, the buyer must provide: (1) a video of the device failing to power on, or showing the specific defect claimed (not just a photo of the device — a photo proves nothing about functionality), AND (2) a photo of the device's serial number to prove it is the same device that was listed. Without these, the dispute cannot be resolved in the buyer's favour on functionality grounds alone.

Ruling Matrix for This Scenario

If listing said "functional" and device is DOA: Tier 1 applies. Full refund of held_amount. Seller's "No Returns" policy is irrelevant. Seller's account is flagged.

If listing accurately described known faults AND buyer confirmed condition grade before purchase: "No Returns / As Is" policy applies. Ruling in seller's favour. The condition grade and the listing photos serve as evidence that the buyer knew what they were getting.

If the evidence is ambiguous (no video proof of defect): Admin may order a partial refund (e.g., 50% of held_amount) as a compromise. Both parties must agree to this in writing via the dispute thread, or admin makes the final unilateral decision.

The Governing Principle — Written into the Platform ToS

"A seller's custom return policy governs voluntary returns. It cannot and does not govern disputes arising from material misrepresentation, counterfeit goods, items damaged in transit, or items that do not match their listed description in a material way. In all such cases, U-Shop's Tier 1 Platform Minimum Buyer Protections apply, and U-Shop's decision is final and binding per the Seller Agreement signed at store creation."
