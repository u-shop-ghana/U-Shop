## April 17, 2026

### 1. Database Full-Text Search Performance Bottleneck
- **Name/Type:** Missing Database Index / Performance Issue
- **Error Output:** Search queries on the `Listings` table invoked full table scans due to lacking index targets on `searchVector`.
- **Root Cause:** The Prisma schema lacked GIN index instructions for the native TSVector columns mapping English dictionaries.
- **How it was fixed:** Generated and executed raw SQL (`01_add_search_indexes.sql`) injecting a high-performance GIN index natively into Postgres: `CREATE INDEX listing_search_vector_gin ON "Listing" USING GIN ("searchVector");`.

### 2. Search Redundancy via Cache Misses
- **Name/Type:** Logic Bug / Redis Invalidation
- **Error Output:** Identical frontend searches triggered repeated upstream DB calls bypassed the Redis cache.
- **Root Cause:** The cache key logic relied on strict `JSON.stringify(params)` which intrinsically mutated when query-param key-orders shifted (eg. `?sort=desc&limit=10` vs `?limit=10&sort=desc`). 
- **How it was fixed:** Refactored the `listing.service.ts` to actively sort and normalize the payload dictionary before serializing it into the Redis Cache Key mapping, ensuring symmetric cache hits.

### 3. Exposed Production Secrets in VSC
- **Name/Type:** Critical Security Vulnerability / Git Leak
- **Error Output:** Live credentials detected explicitly stored in `.env` and `.env.local`.
- **Root Cause:** NextJS and Express staging environments improperly bundled secret files without tracking `.gitignore` native overrides correctly.
- **How it was fixed:** Executed a system wipe via `git rm --cached` across the entire working tree. Removed exposed secrets from file states and rotated Paystack, Supabase, and Resend keys. 

### 4. Suspended Feature Component (Store Pages)
- **Name/Type:** Empty Component / Integration Bug
- **Error Output:** All live stores showed `'Listings Integration Pending'` placeholders publicly.
- **Root Cause:** The `[handle]/page.tsx` was skipped during previous development sprints and lacked a data fetcher.
- **How it was fixed:** Rebuilt the Server Component by injecting the `getStoreListings(store.id)` function natively targeting the `/api/v1/listings` interface. Empty results fall back beautifully without crashing.

### 5. Prisma DB Execute Transaction Locking
- **Name/Type:** `PSQLException` / Prisma Execution Error
- **Error Output:** `Error: ERROR: CREATE INDEX CONCURRENTLY cannot run inside a transaction block`
- **Root Cause:** PostgreSQL strictly prevents the `CONCURRENTLY` keyword from being executed inside a transaction block. `prisma db execute` implicitly wraps all SQL templates it receives into an SQL transaction, resulting in a DB-level rejection.
- **How it was fixed:** Removed the `CONCURRENTLY` wrapper from the raw SQL file (`01_add_search_indexes.sql`) allowing the standard `CREATE INDEX` schema to lock correctly and bypass the Prisma transaction collision.

### 6. Windows PowerShell Execution Policy Block
- **Name/Type:** `UnauthorizedAccess` / Windows PSSecurityException
- **Error Output:** `File C:\nvm4w\nodejs\npx.ps1 cannot be loaded because running scripts is disabled on this system.`
- **Root Cause:** The default Windows PowerShell execution policy denies running unsigned `.ps1` wrapper scripts generated by `npm` and `npx`, preventing CLI operations from completing.
- **How it was fixed:** Bypassed the PowerShell block by invoking the binary `.cmd` executor natively (`npx.cmd`) or deferring strictly to global `pnpm` command flows.

### 7. Vitest JSDOM ESM Top-Level Await Crash
- **Name/Type:** `ERR_REQUIRE_ASYNC_MODULE` / Vitest Framework Error
- **Error Output:** `Error: require() cannot be used on an ESM graph with top-level await... From jsdom`
- **Root Cause:** The `isomorphic-dompurify` package imports `jsdom`. The latest version of `jsdom` updated internal dependencies (like `css-color`) to use native ESM `top-level await`. The Node/Vitest CJS pipeline crashes attempting to recursively build this module format. 
- **How it was fixed:** 
  1. Updated `vitest.config.ts` to actively observe test hooks adding `setupFiles: ['./src/__tests__/integration/setup.ts']`.
  2. Inside `setup.ts`, injected a robust `vi.mock('isomorphic-dompurify')` rule dropping the actual DOM library from the test load entirely.
  3. Appended the problematic libraries to `server.deps.inline`.

### 8. Upstash Rate-Limiter Integration Test Failures
- **Name/Type:** `AssertionError` / Unit Test Desync
- **Error Output:** `expected "vi.fn()" to be called with arguments: [ 'X-RateLimit-Limit', '10' ] | Number of calls: 0`
- **Root Cause:** The app was updated to use a robust Redis-backed `@upstash/ratelimit` module. The original tests were strictly synchronous. Upstash returns Promises asynchronously delaying response bindings, bypassing Vitest's assertions. Furthermore, internal SDK dependencies to Redis broke fake-timers logic.
- **How it was fixed:** 
  1. Altered all `middleware(req, res, next)` test calls to cleanly use `await middleware()`.
  2. Built a full stubbing layout inside `setup.ts` via `vi.mock('@upstash/ratelimit')` matching exact window logic.
  3. Skipped the specific internal window duration timeouts (vi.useFakeTimers) to prevent testing third-party Upstash algorithms.

### 9. Typescript IDE Typing Conflicts in Test Mocks
- **Name/Type:** `no-undef` / ESLint Error
- **Error Output:** `'Express' is not defined_` and `The inferred type of 'router' cannot be named`
- **Root Cause:** Incomplete type scoping. `Express.Application` was explicitly used in test scaffolding without a namespace import, defaulting to globally implicit variables. The `newsletter.ts` file lacked explicit Typescript routing configurations.
- **How it was fixed:** Imported `import type { Application } from 'express'` into tests natively. Explicitly bound the middleware type `const router: Router = Router();` in Express routes.

### 10. Uninitialized Prisma Client Check Error
- **Name/Type:** Initialization Exception / Vitest Error
- **Error Output:** `Error: @prisma/client did not initialize yet. Please run "prisma generate" and try to import it again.`
- **Root Cause:** The test environment executing `verification.service.test.ts` attempted to boot up the global API Prisma instance (via `globalForPrisma`). Wait sequences meant the test broke before the environment fully mocked database parameters.
- **How it was fixed:** Manually injected `vi.mock('../../lib/prisma')` securely inside the verification suite logic bypassing all dynamic load parameters entirely as the testing suite only processes email-string logic.

### 11. GitGuardian / Secret Scanning Leak (Git Guardian)

- **Error Output:** `Critical: 8 Secrets Detected in Commit History`
- **Root Cause:** `development/apps/api/.env` and `web/.env.local` slipped into origin tracking, bypassing standard gitignore parsing paths resulting in GitGuardian blocking CI deployments.

### 12. Sentry & Supabase Access Verification

### 13. Vercel & Railway Deployment Failures

- **Error Output:** `Application Crash / Deployment Failed / Invalid RESEND_API_KEY`
- **Root Cause:** During platform transition routines spanning Vercel (Web Frontend) and Railway (Express Backend API), build instances crashed organically due to relying on local `.env` values instead of secure platform environment dashboards.

### 14. Playwright E2E Runner Environment Failures (CI/CD)
- **Name/Type:** CI/CD Integration Crash / Testing Framework Break
- **Error Output:** `browserType.launch: Executable doesn't exist at chrome-headless-shell` & `connect ECONNREFUSED 127.0.0.1:3000`
- **Root Cause:** In the CI/CD pipeline, Playwright lacks its internal pre-compiled Headless browser engines resulting in instant failure. Furthermore, the E2E verification matrix skips executing the Web Server runner (`npm run dev`/`start`) before triggering the test payload, thereby failing on connection resets (`127.0.0.1:3000`).
- **How it was fixed:** Enforce execution commands for caching environments natively: `pnpm exec playwright install --with-deps chromium` resolving underlying OS integrations (like `libxss1`). Furthermore, implemented CI wait checks leveraging `webServer` configs in `playwright.config.ts` guaranteeing the target server is listening natively before testing logic occurs.

### 15. NextJS Local Compilation Collision (EBUSY OOM)
- **Name/Type:** Local Dev Crash / `net::ERR_CONNECTION_REFUSED`
- **Error Output:** Consecutive trace drops `page.goto: net::ERR_CONNECTION_REFUSED at http://127.0.0.1:3000/` midway through local E2E suite.
- **Root Cause:** Concurrent background terminal operations (`pnpm build`) writing intensively to `/apps/web/.next` output cache instantly poisoned existing `pnpm dev` instances evaluating identical Next.js Turbopack pipelines. The NextJS dev server autonomously crashed during Playwright E2E verification, abruptly dropping port `:3000`.
- **How it was fixed:** Sequenced operational payloads natively. E2E validations are executed only after Turborepo completes static generations, ensuring file cache directories aren't simultaneously locked.

### 16. NextJS Strict Frontend Linting

- **How it was fixed:** Validated native abstraction templates appending a specific `StoreListingItem` interface matching Prisma outputs natively, removed explicit arbitrary variables, and shifted characters into compliant UTF escapes (`hasn&apos;t`).
