Section 01

Feature Overview & State Machine

1.1 What Is This Feature & Why Does It Exist?

Imagine you're a first-time buyer on U-Shop. You find a great laptop listed by a seller you've never met. You pay GHS 1,500 — and then what? Do you just hope the seller sends it? That's the trust problem every marketplace in the world has to solve, and it's why escrow exists.

Escrow is a holding pattern for money. Instead of sending your GHS 1,500 directly to the seller the moment you pay, the platform holds it in a secure middle ground. The seller knows the money is real and committed. The buyer knows the seller won't vanish with it. Only after the buyer says "yes, I received the item and I'm satisfied" does the money move to the seller.

Why Not Just Use a Payment Gateway's Built-in Escrow?

Paystack doesn't offer native escrow as a product feature in Ghana. This is why we build it ourselves — not by holding cash in a vault, but by using Paystack as a "float account". When a buyer pays, 100% of the funds land in your platform's Paystack balance. Our database tracks what belongs to whom through a ledger. When it's time to pay the seller, we initiate a Paystack Transfer from our balance to their MoMo. The database is the escrow system. Paystack is just the payment rails.

1.2 The Order State Machine

A "state machine" is just a formal way of saying: an order can only be in one defined state at a time, and can only move to the next state in a valid direction. You can't go from PENDING_PAYMENT directly to COMPLETED. There's a strict path. This prevents bugs and financial inconsistencies.

01
PENDING_PAYMENT Trigger: Buyer clicks "Buy Now"

The order record exists in our database, but no money has moved. Express has called Paystack's /transaction/initialize and received a checkout URL. The buyer has been redirected to Paystack's payment page but has not completed payment yet. This order will be garbage-collected if payment never comes (via a cleanup cron job).

02
PAYMENT_RECEIVED Trigger: Paystack fires the charge.success webhook

The buyer's MoMo or card payment was successful. GHS 1,500 has landed in the platform's Paystack account balance. The Escrow record is created in our database with status: HOLDING. The listing is marked SOLD. The 7-day auto-release timer starts here. This is the "money in the vault" moment.

03
PROCESSING / IN_TRANSIT Trigger: Seller updates the order manually

The seller has acknowledged the order and is preparing it (PROCESSING), or has handed it off / arranged a meetup (IN_TRANSIT). These are informational states for buyer visibility — they do not affect the escrow funds. The money is still locked.

04
RELEASING_FUNDS Trigger: Buyer clicks "I've Received My Item" OR 7-day timer fires

This is a critical intermediate state we set immediately when the buyer confirms receipt. It acts as a distributed lock — it tells the system "a payout is in flight, don't touch this escrow." We set this state before calling Paystack's API. If the server crashes mid-transfer, this state tells us exactly what happened when we investigate.

05
COMPLETED Trigger: Paystack fires the transfer.success webhook

The Paystack transfer to the seller's MoMo landed successfully. The Escrow record is set to RELEASED. The seller's in-platform wallet balance is credited. The order is fully done. This state is terminal — nothing can change it.

⚠
DISPUTED Trigger: Buyer clicks "Open a Dispute" before confirming receipt

The buyer has a problem — the item never arrived, is damaged, or isn't as described. The escrow is frozen — no auto-release can fire while a dispute is OPEN or UNDER_REVIEW. An admin must resolve it before any money moves. This is the buyer's safety valve and the most important escape hatch in the flow.

The Golden Rule of State Machines

Every state transition should be a reaction to an external event, not something Express decides to do on its own schedule. The events that drive this machine are: buyer pays (Paystack webhook), buyer confirms (API call), transfer lands (Paystack webhook), buyer disputes (API call). Your Express code is an event handler, not a scheduler. This mental model will save you from dozens of race condition bugs.

Section 02

Frontend Implementation (Next.js 14)

2.1 UI Components Needed

The buyer interacts with the escrow flow from a single page: the Order Detail page. Here's what it needs to contain and the reasoning behind each element.

📱 Order Detail Page — UI Mock (Order in PAYMENT_RECEIVED state)
Order #ORD-2024-00142
⏳ Awaiting Delivery
Apple MacBook Air M2 (2022)
Seller janes-tech-store
Amount Paid GHS 1,500.00
Escrow Status 🔒 Held — Funds Protected
Auto-Release In 4 days, 6 hours
⚠ Only click "Item Received" once you have physically inspected the item and are satisfied. This action is irreversible and will immediately initiate payment to the seller.

Three things are deliberately designed into this UI:

1. The auto-release countdown. The buyer must feel urgency. If they forget to confirm for 7 days, the seller gets paid anyway. Showing the timer creates accountability and prevents "I forgot to confirm" disputes.

2. The irreversibility warning. This is not just user experience — it is legal protection. If a buyer clicks "received" and then claims they didn't, this warning (plus your server logs) shows they were explicitly informed of the consequence.

3. The dispute button is equal prominence, not hidden. If a buyer wants to open a dispute they should find it easily. Hiding it makes them more likely to call their bank and initiate a chargeback — which is far worse for the platform.

2.2 The "Confirm Receipt" — Exact React Logic

app/orders/[orderId]/page.tsx — The Confirm Receipt logic (annotated)
// This is a CLIENT component because it has interactive state (loading, errors).
// The page itself can be a Server Component that passes the order as a prop.
'use client'

import { useState } from 'react'
import { useRouter } from 'next/navigation'
import { createBrowserClient } from '@supabase/ssr'

// ─── Types ───────────────────────────────────────────────────────────────────
type ConfirmState = 'idle' | 'confirming' | 'releasing' | 'done' | 'error'

interface Props {
  order: {
    id: string
    status: string
    amount: string
    sellerName: string
    listingTitle: string
    autoReleaseAt: string
  }
}

export function ConfirmReceiptSection({ order }: Props) {
  // ── Local state ───────────────────────────────────────────────────────────
  // 'idle'        — nothing happening, show the button
  // 'confirming'  — we're showing the "are you sure?" dialog
  // 'releasing'   — API call in flight, button is disabled and shows spinner
  // 'done'        — API call succeeded, show success state
  // 'error'       — API call failed, show error message and allow retry
  const [state, setState] = useState<ConfirmState>('idle')
  const [errorMessage, setErrorMessage] = useState<string | null>(null)
  const [showConfirmDialog, setShowConfirmDialog] = useState(false)
  const router = useRouter()

  // ── Step 1: First click — show confirmation dialog ────────────────────────
  // We do NOT call the API on first click. We make the user confirm twice.
  // This prevents rage-clicks and accidental releases.
  function handleInitialClick() {
    setShowConfirmDialog(true)
    setState('confirming')
  }

  // ── Step 2: User confirms in the dialog — NOW we call the API ─────────────
  async function handleConfirmedRelease() {
    setShowConfirmDialog(false)
    setState('releasing')
    setErrorMessage(null)

    try {
      // ── Get the current Supabase session token ─────────────────────────────
      // IMPORTANT: We fetch this fresh at call time, not once at component mount.
      // Tokens expire after 1 hour. A user who's been on the page for a long time
      // could have an expired token if we cached it. This prevents a 401 error.
      const supabase = createBrowserClient(
        process.env.NEXT_PUBLIC_SUPABASE_URL!,
        process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
      )
      const { data: { session } } = await supabase.auth.getSession()

      if (!session) {
        // User got logged out while on the page — send them to login
        router.push(`/login?redirect=/orders/${order.id}`)
        return
      }

      // ── Call the Express release-escrow endpoint ───────────────────────────
      const response = await fetch(
        `${process.env.NEXT_PUBLIC_API_URL}/orders/${order.id}/release-escrow`,
        {
          method: 'POST',
          headers: {
            'Content-Type': 'application/json',
            // This is the Supabase JWT — Express will verify it server-side
            'Authorization': `Bearer ${session.access_token}`,
          },
          // Optional: send a rating with the confirm action
          body: JSON.stringify({ buyerRating: 5 }),
        }
      )

      const result = await response.json()

      // ── Handle non-2xx responses ───────────────────────────────────────────
      if (!response.ok) {
        // The Express server returns structured errors: { success: false, error: { code, message } }
        // We map specific error codes to user-friendly messages
        const code = result.error?.code

        if (code === 'ESCROW_ALREADY_RELEASED') {
          // This can happen if the user somehow clicked twice (idempotency guard)
          // Refresh the page to show the completed state
          router.refresh()
          return
        }

        if (code === 'NO_PAYOUT_ACCOUNT') {
          setErrorMessage(
            'The seller has not yet registered a payout account. ' +
            'Your funds are safe. Please contact support.'
          )
        } else {
          // Generic error — use the message from Express
          setErrorMessage(result.error?.message ?? 'Something went wrong. Please try again.')
        }

        setState('error')
        return
      }

      // ── Success ────────────────────────────────────────────────────────────
      // The transfer is INITIATED (not yet landed — that takes seconds to minutes).
      // We show a success state and refresh the page data.
      setState('done')
      // router.refresh() forces Next.js to re-fetch the server component data
      // so the order status updates to COMPLETED on screen.
      router.refresh()

    } catch (networkError) {
      // Network-level failure (fetch itself threw) — the user has no internet
      // or the Express server is completely unreachable.
      // IMPORTANT: We do NOT know if the server processed the request.
      // We tell the user to check their order status before retrying.
      setState('error')
      setErrorMessage(
        'Network error. Please check your connection and refresh the page ' +
        'before trying again to avoid a duplicate release.'
      )
    }
  }

  // ── Render ────────────────────────────────────────────────────────────────
  if (state === 'done') {
    return <SuccessBanner message="Payment released! The seller is being paid now." />
  }

  return (
    <div>
      {errorMessage && <ErrorBanner message={errorMessage} onDismiss={() => setState('idle')} />}

      {showConfirmDialog && (
        <ConfirmDialog
          title="Are you sure you received this item?"
          body="This will immediately release GHS {order.amount} to {order.sellerName}. This cannot be undone."
          confirmLabel="Yes, I've received it — Release Payment"
          cancelLabel="No, Go Back"
          onConfirm={handleConfirmedRelease}
          onCancel={() => { setShowConfirmDialog(false); setState('idle') }}
        />
      )}

      <button
        onClick={handleInitialClick}
        disabled={state === 'releasing'}
        // Disable button while release is in flight — prevents double submission
      >
        {state === 'releasing'
          ? '⏳ Releasing funds…'
          : '✓ I\'ve Received My Item — Release Payment'
        }
      </button>
    </div>
  )
}

2.3 Why We Use a Confirmation Dialog (Double-Click Protection)

Notice the two-step interaction: first click opens a dialog, second click fires the API. This is not over-engineering — it is essential. The escrow release is one-directional and immediate. There is no "undo" button. A buyer who accidentally clicks the button and instantly initiates a transfer to a seller has no recourse without manual admin intervention. The confirmation dialog is the cheapest possible insurance policy.

ℹ️

Why router.refresh() and not router.push()? In Next.js App Router, router.refresh() tells the server to re-run all Server Components on the current page and send fresh data — without losing the user's scroll position or client state. It's the correct way to "sync" the page after a mutation in the App Router paradigm.

Section 03

Backend Implementation (Express.js & Prisma)

This is the most consequential code in the entire platform. It touches real money. Every line needs to be intentional. We'll build it in layers: security guards first, then the database transaction, then the Paystack API call.

The Core Design Principle for this Endpoint

Think of this handler as a three-door lock. Door 1: Is the person who they claim to be? Door 2: Are they allowed to do this specific thing? Door 3: Is the system in the right state to do it? Only after all three doors open do we touch the database, and only after the database is committed do we talk to Paystack. Never call an external API inside a database transaction that you can't roll back.

3.1 Security Guards — The Four Checks Before Any Action

G1

Is the caller authenticated? → 401 MISSING_TOKEN

The authenticate middleware runs first. It reads the Authorization: Bearer <token> header, verifies it with Supabase's service role SDK, and attaches req.user. If anything is wrong — no header, expired token, tampered token — the request is dead before it reaches the route handler. No route logic runs.

G2

Does the order exist? → 404 ORDER_NOT_FOUND

We fetch the order from the database using the :orderId path parameter. If it doesn't exist, return 404. This also protects against probing attacks where a malicious user guesses order IDs — they just get 404s.

G3

Is the caller the buyer of THIS order? → 403 NOT_YOUR_ORDER

Compare order.buyerId (from the database) against req.user.id (from the verified JWT). If they don't match, return 403. A seller cannot release their own escrow. Another buyer cannot release someone else's order. This check is the most critical — without it, any authenticated user could release any escrow.

G4

Is the order in a releasable state? → 400 INVALID_ORDER_STATE

The order's status must be one of PAYMENT_RECEIVED, PROCESSING, or IN_TRANSIT. If it's already COMPLETED, RELEASED, or DISPUTED, we return an appropriate error. This is the idempotency guard at the business logic level — the Prisma transaction is the idempotency guard at the database level (belt-and-suspenders).

3.2 The Prisma Transaction — The Most Important Code Block

Here is the most important concept in this entire spec. Read it carefully.

Imagine two operations need to happen: (A) update the database to say the escrow is releasing, and (B) call Paystack to initiate the transfer. What happens if A succeeds but B fails? The database says "released" but the seller never got paid. The money is lost — or rather, it's stranded in your Paystack balance with no record of what it was for.

The solution is: do A inside a transaction, do B outside it, and use an intermediate state to bridge the gap.

routes/orders.ts — POST /api/orders/:orderId/release-escrow (fully annotated)
import { Router } from 'express'
import { Decimal } from '@prisma/client/runtime/library'
import { authenticate } from '../middleware/authenticate'
import { PaystackService } from '../services/paystack'
import { buildSuccess, buildError } from '../utils/response'
import { prisma } from '../lib/prisma'

const router = Router()
const PLATFORM_FEE_RATE = new Decimal('0.05')  // 5% platform cut

// All order routes require authentication
router.use(authenticate)

router.post('/:orderId/release-escrow', async (req, res, next) => {
  try {
    const { orderId } = req.params
    const callerId = req.user!.id  // Set by authenticate middleware

    // ═══════════════════════════════════════════════════════════════════════
    // PHASE 1: SECURITY GUARDS — All checks before ANY database writes
    // ═══════════════════════════════════════════════════════════════════════

    // Fetch the order with everything we need for validation
    const order = await prisma.order.findUnique({
      where: { id: orderId },
      include: {
        escrow: true,
        seller: {
          include: { payoutRecipient: true }
        }
      }
    })

    // Guard 2: Order must exist
    if (!order) {
      return res.status(404).json(buildError('ORDER_NOT_FOUND', 'Order does not exist'))
    }

    // Guard 3: The person calling MUST be the buyer of THIS specific order
    // We compare the DB record's buyerId against the JWT-verified user ID.
    // This cannot be faked because req.user.id comes from a Supabase-verified token.
    if (order.buyerId !== callerId) {
      return res.status(403).json(buildError('NOT_YOUR_ORDER',
        'You do not have permission to release this escrow'))
    }

    // Guard 4a: Order must be in a releasable state
    const releasableStates = ['PAYMENT_RECEIVED', 'PROCESSING', 'IN_TRANSIT']
    if (!releasableStates.includes(order.status)) {
      // Handle the special case: already released (idempotency for client UI)
      if (order.status === 'COMPLETED') {
        return res.status(409).json(buildError('ESCROW_ALREADY_RELEASED',
          'Payment has already been released for this order'))
      }
      if (order.status === 'DISPUTED') {
        return res.status(409).json(buildError('ORDER_DISPUTED',
          'This order has an active dispute. Funds are frozen.'))
      }
      return res.status(400).json(buildError('INVALID_ORDER_STATE',
        `Order cannot be released in its current state: ${order.status}`))
    }

    // Guard 4b: Escrow must exist and be in HOLDING state
    if (!order.escrow || order.escrow.status !== 'HOLDING') {
      return res.status(400).json(buildError('ESCROW_NOT_HOLDING',
        'Escrow funds are not in a releasable state'))
    }

    // Guard 4c: Seller must have a registered payout account
    if (!order.seller.payoutRecipient?.recipientCode) {
      return res.status(400).json(buildError('NO_PAYOUT_ACCOUNT',
        'Seller has not registered a payout account. Please contact support.'))
    }

    // ═══════════════════════════════════════════════════════════════════════
    // PHASE 2: PRISMA TRANSACTION — Atomically claim the escrow
    // ═══════════════════════════════════════════════════════════════════════
    // Why a transaction? We need to update MULTIPLE tables together.
    // If any one update fails, ALL of them roll back automatically.
    // This prevents partial state: e.g., order marked COMPLETED but
    // escrow still showing HOLDING, or Payout record never created.
    //
    // WHY set escrow to RELEASING (not RELEASED) here?
    // Because the Paystack transfer hasn't happened yet. RELEASING is
    // a "claim" — it tells any other concurrent request "this is taken,
    // back off." The WHERE clause { status: 'HOLDING' } is the atomic lock.
    // If two requests hit this endpoint at the same time, only ONE will
    // find the escrow in HOLDING state. The second will throw, and we
    // catch it below and return a 409.

    const gross = new Decimal(order.escrow.amount)
    const platformFee = gross.mul(PLATFORM_FEE_RATE).toDecimalPlaces(2)
    const sellerNet = gross.sub(platformFee)
    // Paystack requires amounts in kobo (smallest currency unit). GHS 450.00 = 45000 kobo.
    const koboAmount = sellerNet.mul(100).toDecimalPlaces(0).toNumber()

    let newPayoutId: string

    try {
      const result = await prisma.$transaction(async (tx) => {

        // 1. Atomically flip escrow from HOLDING → RELEASING
        // The { where: { status: 'HOLDING' } } clause is the race condition guard.
        // Prisma throws P2025 (record not found) if the escrow is NOT in HOLDING state,
        // which rolls back the entire transaction automatically.
        const claimedEscrow = await tx.escrow.update({
          where: {
            orderId: orderId,
            status: 'HOLDING'   // ← This is the atomic lock. Fail if not HOLDING.
          },
          data: { status: 'RELEASING' }
        })

        // 2. Update order status to show buyer has confirmed
        await tx.order.update({
          where: { id: orderId },
          data: {
            status: 'RELEASING_FUNDS',
            confirmedAt: new Date(),
            buyerRating: req.body.buyerRating ?? null,
          }
        })

        // 3. Create a Payout record BEFORE calling Paystack
        // This is our audit trail. Even if Paystack fails, we have a record
        // of the attempt with PENDING status. We'll store the transferCode here
        // once Paystack responds.
        const payout = await tx.payout.create({
          data: {
            sellerId: order.sellerId,
            orderId: orderId,
            amount: sellerNet,
            platformFee: platformFee,
            status: 'PENDING',  // Will become PROCESSING after Paystack call
          }
        })

        return { claimedEscrow, payout }
      })

      newPayoutId = result.payout.id

    } catch (txError: any) {
      // P2025 = Prisma "record to update not found"
      // This means the escrow was NOT in HOLDING state (already released, or race condition)
      if (txError.code === 'P2025') {
        return res.status(409).json(buildError('ESCROW_ALREADY_RELEASED',
          'This escrow has already been released or is currently being processed'))
      }
      throw txError  // Re-throw unknown errors to the global handler
    }

    // ═══════════════════════════════════════════════════════════════════════
    // PHASE 3: PAYSTACK TRANSFER — Called AFTER the DB transaction commits
    // ═══════════════════════════════════════════════════════════════════════
    // We are NOW outside the Prisma transaction. The transaction committed above.
    // The escrow is RELEASING. The Payout record exists with status PENDING.
    // NOW we call Paystack. If THIS call fails, the system is in a recoverable
    // state: a cron job can find all Payout records with status PENDING and retry.

    let transferCode: string

    try {
      const transfer = await PaystackService.initiateTransfer({
        source: 'balance',
        amount: koboAmount,
        recipient: order.seller.payoutRecipient!.recipientCode,
        reason: `U-Shop payout - Order ${orderId}`,
        reference: `payout_${newPayoutId}`,  // Idempotency key for Paystack
      })

      transferCode = transfer.transfer_code

      // Store the transferCode on the Payout record.
      // This is CRITICAL. When Paystack fires transfer.success/failed webhooks,
      // they include transfer_code. Without storing this now, we can't match
      // the webhook event back to the correct Payout record.
      await prisma.payout.update({
        where: { id: newPayoutId },
        data: { transferCode, status: 'PROCESSING' }
      })

    } catch (paystackError) {
      // ─────────────────────────────────────────────────────────────────────
      // PAYSTACK CALL FAILED. The database is in RELEASING state.
      // The money is NOT lost. It's still in the platform's Paystack balance.
      // The Payout record has status PENDING — this is our recovery breadcrumb.
      // A cron job will find this PENDING payout and retry the transfer.
      // See Section 4.3 for the full recovery playbook.
      // ─────────────────────────────────────────────────────────────────────
      console.error('[PAYSTACK TRANSFER FAILED]', paystackError)

      // Log to Sentry for immediate alerting
      Sentry.captureException(paystackError, { extra: { orderId, newPayoutId } })

      // We still return a 200 to the client — from THEIR perspective, they
      // confirmed receipt successfully. The payment will arrive once the retry fires.
      // Do NOT return a 500 here — that would confuse the buyer and potentially
      // cause them to click the button again.
      return res.status(200).json(buildSuccess({
        orderId,
        message: 'Receipt confirmed. Payment to seller is being processed and may take a few minutes.',
        status: 'RELEASING_FUNDS'
      }))
    }

    // ─── All good — transfer initiated ────────────────────────────────────────
    return res.status(200).json(buildSuccess({
      orderId,
      orderStatus: 'RELEASING_FUNDS',
      escrowStatus: 'RELEASING',
      transferCode,
      sellerNet: sellerNet.toString(),
      platformFee: platformFee.toString(),
      message: 'Transfer initiated. Seller will receive payment shortly.',
    }))

  } catch (err) {
    next(err)  // Hand to global error handler
  }
})

export const ordersRouter = router

3.3 The Paystack Transfer Service

Keeping the Paystack API call in its own service module is important for testability and reusability. The retry cron job will call this same function.

services/paystack.ts — initiateTransfer method
const PAYSTACK_BASE = 'https://api.paystack.co'
const PAYSTACK_SECRET = process.env.PAYSTACK_SECRET_KEY!  // Express env only, NEVER frontend

interface TransferParams {
  source: 'balance'     // Always 'balance' — meaning "from our Paystack account"
  amount: number        // In KOBO (integer). GHS 100 = 10000 kobo.
  recipient: string    // The seller's Paystack recipient_code (e.g. "RCP_hx66...")
  reason: string        // Shows on seller's MoMo statement
  reference: string    // Our unique reference — Paystack uses this for idempotency
}

export const PaystackService = {

  async initiateTransfer(params: TransferParams) {
    const response = await fetch(`${PAYSTACK_BASE}/transfer`, {
      method: 'POST',
      headers: {
        'Authorization': `Bearer ${PAYSTACK_SECRET}`,
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({
        source: params.source,
        amount: params.amount,
        recipient: params.recipient,
        reason: params.reason,
        reference: params.reference,
        currency: 'GHS',
      }),
    })

    const data = await response.json()

    if (!response.ok || !data.status) {
      // Paystack returned an error response — throw so the caller can handle it
      throw new Error(`Paystack transfer failed: ${data.message ?? JSON.stringify(data)}`)
    }

    // Returns { transfer_code: "TRF_xxx", status: "pending", amount: 45000, ... }
    return data.data
  },

  async verifyTransaction(reference: string) {
    const response = await fetch(
      `${PAYSTACK_BASE}/transaction/verify/${reference}`,
      { headers: { Authorization: `Bearer ${PAYSTACK_SECRET}` } }
    )
    const data = await response.json()
    return data.data
  }
}

3.4 The Webhook Handler — The Final Chapter of the Payout

When you call Paystack's Transfer API, the transfer does not complete instantly. Paystack processes it and then fires a webhook to your server when it's done. This is how you know to mark the order as truly COMPLETED.

routes/webhooks.ts — handleTransferSuccess (called from the verified webhook handler)
// This function is called from the main webhook handler AFTER signature verification
async function handleTransferSuccess(data: PaystackTransferData) {
  // 1. Find the Payout record using the transfer_code Paystack sent us
  // This is why storing transferCode after initiating was critical.
  const payout = await prisma.payout.findUniqueOrThrow({
    where: { transferCode: data.transfer_code },
    include: { order: { select: { id: true, sellerId: true } } }
  })

  await prisma.$transaction(async (tx) => {
    // 2. Mark the escrow as RELEASED — funds have left the platform
    await tx.escrow.update({
      where: { orderId: payout.orderId },
      data: { status: 'RELEASED', releasedAt: new Date() }
    })

    // 3. Mark the Payout as COMPLETED
    await tx.payout.update({
      where: { id: payout.id },
      data: { status: 'COMPLETED', completedAt: new Date() }
    })

    // 4. Mark the Order as fully COMPLETED
    await tx.order.update({
      where: { id: payout.orderId },
      data: { status: 'COMPLETED' }
    })

    // 5. Credit the seller's in-platform wallet for record-keeping
    // (The real payout happened via MoMo — this is the ledger entry)
    const amountGHS = new Decimal(data.amount).div(100)
    await tx.wallet.update({
      where: { userId: payout.order.sellerId },
      data: { lifetimeEarnings: { increment: amountGHS } }
    })
  })

  // 6. Notify seller that they've been paid
  await sendSellerPaidNotification(payout.orderId)
}
Section 04

Security & The Failure Playbook

This section answers the hardest questions: what happens when things go wrong? A system that works perfectly in the happy path but fails dangerously in edge cases is not production-ready. Let's address each failure mode.

4.1 Idempotency — Preventing Double-Release

The problem: A user clicks "Confirm Receipt." The API call fires. The network is slow. They click again. Now two requests are racing to release the same escrow. If the second one succeeds, GHS 1,500 goes to the seller twice from your Paystack balance.

The solution is a three-layer defense:

L1

Frontend: Disable the button while in-flight

The React component sets state = 'releasing' immediately on first click, which disables the button. This stops accidental double-clicks before any network request is even made. It's fast and costs nothing — but it's only a UI guard. A determined user (or a script) can still bypass it.

Why this alone isn't enough

The frontend can always be bypassed. A user can open the browser devtools and fire the fetch manually, or an attacker can call the endpoint directly with curl. Frontend guards are UX, not security.

L2

Backend: State guard check before the transaction

Express checks order.status before doing anything. If it's already RELEASING_FUNDS, COMPLETED, or the escrow is not in HOLDING state, we return a 409 immediately. This stops the second request that arrives after the first transaction commits.

L3

Database: Atomic WHERE clause in the Prisma update

This is the bulletproof layer. The Prisma transaction does: escrow.update({ where: { orderId, status: 'HOLDING' } }). If two requests arrive at the exact same millisecond (a true race condition that layers 1 and 2 cannot catch), the database will serialize them. Only one will find the escrow in HOLDING state. The other will get a P2025 error, the transaction rolls back, and we return a 409. This is a database-level lock — it cannot be beaten.

Why we also send a reference to Paystack

We pass reference: payout_${newPayoutId} to Paystack's transfer API. Paystack's own system uses this to deduplicate — if the same reference arrives twice, Paystack only processes it once. This is Paystack's idempotency layer, which protects against our server calling the API twice (e.g., from a retry cron job and the original request both succeeding).

4.2 Ownership Verification — Stopping Unauthorized Releases

The threat: User A (who owns Order #001) tries to release User B's escrow (Order #002) by sending POST /orders/order-002-id/release-escrow with their own valid JWT.

Why This Is Critical

A Supabase JWT only proves that the person is who they say they are. It does NOT prove they are allowed to perform an action on a specific resource. Authentication (who are you?) and authorization (are you allowed to do this?) are two completely separate concepts. Your middleware handles authentication. Your route handler must handle authorization.

The ownership check — the most critical guard in the route
// req.user.id comes from Supabase's server-verified JWT — this cannot be faked
// order.buyerId comes from YOUR database — this is the ground truth
// If they don't match, the request is unauthorized. No exceptions.

if (order.buyerId !== req.user!.id) {
  // Important: Return 403 (Forbidden), NOT 404 (Not Found).
  // Some APIs return 404 to hide that the resource exists at all (security through obscurity).
  // For an escrow endpoint, a 403 is more honest and traceable in your logs.
  // The attacker already knows the order exists — they guessed the ID.
  return res.status(403).json(buildError(
    'NOT_YOUR_ORDER',
    'You do not have permission to release this escrow'
  ))
}

// Also: Make sure we check the BUYER, not just any authenticated user.
// The SELLER should NOT be able to release their own escrow.
// (A seller releasing their own escrow would be paying themselves before delivery.)
// Since we check order.buyerId === req.user.id, and the seller is order.sellerId,
// a seller calling this endpoint will fail this check automatically. ✓

4.3 The Paystack Failure Playbook — When the Transfer API is Down

The scenario: The buyer clicks "Confirm Receipt." Our Prisma transaction commits perfectly — the escrow is in RELEASING state, the Payout record exists with PENDING status. Then we call Paystack's Transfer API... and it times out. Paystack is having an incident.

🔴

Failure: Paystack Transfer API Returns Error / Times Out

Recoverable
What happens immediately

The catch(paystackError) block fires. We log the error to Sentry (which alerts the team). We return a friendly 200 to the buyer: "Payment to seller is being processed and may take a few minutes." We do NOT return a 500 — that would confuse the buyer and might make them click again.

System state at this point
Table / FieldValueMeaning
Order.statusRELEASING_FUNDSBuyer confirmed. Transfer in flight (or failed).
Escrow.statusRELEASINGClaimed. Will not be double-released.
Payout.statusPENDINGTransfer was never successfully initiated. Needs retry.
Payout.transferCodenullWe never got a code from Paystack. Retry needed.
How the system recovers

A cron job runs every 5 minutes and queries for any Payout records with status PENDING that are more than 2 minutes old. For each one, it retries the Paystack transfer call. It passes the same reference: payout_${payoutId} — so if Paystack somehow received the first call but failed to respond, the duplicate will be rejected by Paystack's idempotency and won't double-pay the seller.

jobs/retryPendingPayouts.ts — the recovery cron job
// Run this every 5 minutes via node-cron or BullMQ
export async function retryPendingPayouts() {
  const stuckPayouts = await prisma.payout.findMany({
    where: {
      status: 'PENDING',
      createdAt: { lte: new Date(Date.now() - 2 * 60 * 1000) }  // > 2 mins old
    },
    include: {
      seller: { include: { payoutRecipient: true } }
    }
  })

  for (const payout of stuckPayouts) {
    try {
      // The reference is the same as before — Paystack deduplicates it
      const transfer = await PaystackService.initiateTransfer({
        source: 'balance',
        amount: payout.amount.mul(100).toDecimalPlaces(0).toNumber(),
        recipient: payout.seller.payoutRecipient!.recipientCode,
        reason: `U-Shop payout retry - Order ${payout.orderId}`,
        reference: `payout_${payout.id}`,  // Same reference as original attempt
      })

      await prisma.payout.update({
        where: { id: payout.id },
        data: { transferCode: transfer.transfer_code, status: 'PROCESSING' }
      })

      console.log(`[RETRY SUCCESS] Payout ${payout.id} initiated`)

    } catch (err) {
      // Still failing — log and try again on the next cron run
      console.error(`[RETRY FAILED] Payout ${payout.id}:`, err)
      Sentry.captureException(err, { extra: { payoutId: payout.id } })
    }
  }
}
🟡

Failure: transfer.failed Webhook — Transfer Bounced at MoMo Level

Requires Admin

This happens when Paystack accepted the transfer, gave us a transfer_code, but the underlying MoMo network rejected it (e.g., the seller's number is not registered on the network they claimed, or the account is inactive). Paystack fires a transfer.failed webhook.

Recovery action

In the handleTransferFailed webhook handler: set the Payout status to FAILED, revert the Escrow status back to HOLDING, and set the Order status back to RELEASING_FUNDS (or a new PAYOUT_FAILED state if you add it). Send an alert to your admin dashboard. An admin must contact the seller to update their payout account, then manually trigger a retry from the admin panel via POST /api/admin/payouts/:id/retry.

4.4 The 7-Day Auto-Release — The Buyer Who Never Responds

What happens if a buyer receives their item, is happy with it, but simply never clicks "Confirm Receipt"? The seller is stuck waiting indefinitely. This is unfair to honest sellers. The solution is an auto-release timer.

⚠️

The Auto-Release Paradox: The auto-release protects sellers from non-responsive buyers. But it also means a buyer who genuinely hasn't received their item has a 7-day window to open a dispute before the money goes to the seller. This makes aggressive notification (Day 3, Day 5, Day 6, Day 6.5 reminders) a financial safety feature, not just UX polish. A buyer who misses the window due to lack of notification is a platform liability.

jobs/autoReleaseEscrow.ts — the 7-day auto-release cron job
// Run this ONCE per hour (not every minute — it's not time-critical to the second)
export async function processAutoReleases() {
  // Find all escrows where the auto-release timer has passed
  // AND the order hasn't been resolved by the buyer yet
  const expiredEscrows = await prisma.escrow.findMany({
    where: {
      status: 'HOLDING',                                    // Still held
      autoReleaseAt: { lte: new Date() },                   // Timer has elapsed
      order: { status: { notIn: ['DISPUTED', 'CANCELLED'] } }  // Not under dispute
    },
    include: { order: { include: { seller: { include: { payoutRecipient: true } } } } }
  })

  console.log(`[AUTO-RELEASE] Processing ${expiredEscrows.length} expired escrows`)

  for (const escrow of expiredEscrows) {
    try {
      // Reuse the same release logic as the manual endpoint
      // This ensures identical behavior whether triggered by buyer or timer
      await releaseEscrowForOrder(escrow.orderId, { trigger: 'AUTO_RELEASE' })
      console.log(`[AUTO-RELEASE] Released escrow for order ${escrow.orderId}`)
    } catch (err) {
      console.error(`[AUTO-RELEASE] Failed for order ${escrow.orderId}:`, err)
      Sentry.captureException(err, { extra: { orderId: escrow.orderId } })
    }
  }
}

// Shared release logic — called by both the API route and the cron job

  // Extract this to a shared service function so both callers use identical logic
  await EscrowService.release(orderId, { ...meta })
}
Principal Engineer's Summary — The Four Guarantees of This System

A well-built escrow system makes four guarantees. (1) Money can never disappear — it's either in Paystack's balance (platform float) or in the seller's account. The database ledger tracks which. (2) The escrow can only be released once — the atomic WHERE status = 'HOLDING' Prisma update is a database-level lock. (3) Only the buyer of the specific order can release it — the ownership check against the JWT-verified user ID enforces this. (4) Paystack failures are recoverable — the PENDING Payout record is a breadcrumb that the retry cron job follows. If you implement these four guarantees, this feature is production-safe.
