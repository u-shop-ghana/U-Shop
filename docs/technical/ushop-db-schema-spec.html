1.2 Plain-English Relationship Summary

USER relationships

  • 1:0–1A user optionally owns one Store. Buyers may never create a store. Sellers have exactly one.
  • 1:0–1A user optionally has one Wallet. Only created when a user first earns from a sale.
  • 1:manyA user can place many Orders as a buyer.
  • noteThe "4 personas" (Student Buyer, Normal Buyer, Student Seller, Reseller) are NOT separate tables. They're derived from role + verificationStatus. See Section 2b for the full reasoning.

STORE relationships

  • 1:manyA store has many Listings. A listing belongs to exactly one store.
  • 1:manyA store receives many Orders (orders involving its listings).
  • noteThe store's return_policy JSON is the current policy. Listings and Orders each carry their own immutable snapshot of the policy at creation/checkout time.

LISTING relationships

  • many:1A listing belongs to one store and one category.
  • 1:manyA listing can appear in many OrderItems (historical purchases).
  • 1:manyA listing has a message thread (Q&A before purchase).
  • noteprice and unit_price on OrderItem are SEPARATE. If the seller changes the price, old orders are unaffected.

ORDER relationships

  • 1:manyAn order contains many OrderItems (currently 1 item/order, but schema supports carts).
  • 1:1An order has exactly one EscrowLedger. Created when payment is confirmed.
  • 1:0–1An order may have at most one Dispute.
  • 1:manyAn order has a message thread (post-purchase buyer-seller chat).

ESCROW LEDGER relationships

  • 1:1Strictly one-to-one with Order. order_id is a unique foreign key.
  • noteThis table is the financial source of truth. It stores both the Paystack charge reference AND the transfer reference, giving you a complete audit trail from money-in to money-out in a single row.

WALLET relationships

  • 1:1One wallet per user. Created lazily on first sale.
  • 1:manyA wallet has many WalletTransactions (the immutable ledger of every credit and debit).
  • 1:manyA wallet funds many Payouts (each withdrawal request is a separate Payout record).
  • noteavailable_balance should always equal the sum of all WalletTransactions. Periodically reconcile these to catch bugs.

2.1 Generator, Datasource & Enums

Enums are fixed sets of allowed values enforced at the database level. This means if your application code ever tries to insert a typo like "stuudent", PostgreSQL will reject it outright — no silent bad data ever enters the system.

// ── DATASOURCE ────────────────────────────────────────────────────────────────
// "postgresql" tells Prisma we're connecting to a PostgreSQL database (Supabase).
//
// CRITICAL: Two separate URLs are required for Supabase + Prisma:
//
//   DATABASE_URL  → Port 6543 (PgBouncer pooler) — used for ALL application queries.
//                   PgBouncer multiplexes thousands of app connections through ~20
//                   real DB connections. Without it, you'll hit connection limits fast.
//                   The "?pgbouncer=true" flag disables Prisma's prepared statements,
//                   which PgBouncer's transaction mode does not support.
//
//   DIRECT_URL    → Port 5432 (direct connection) — used ONLY by "prisma migrate".
//                   Migrations run DDL statements (CREATE TABLE, ALTER TABLE) that
//                   cannot go through PgBouncer's transaction mode pooling.
//                   This URL is NEVER used by application queries.
datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")   // Port 6543 — pooled, for queries
  directUrl = env("DIRECT_URL")    // Port 5432 — direct, for migrations only
}


// ═══════════════════════════════════════════════════════════════════════════════
// SECTION A — ENUMS
// ═══════════════════════════════════════════════════════════════════════════════
//
// Enums live in PostgreSQL as constraint types. If any application code attempts
// to write a value not in the enum, the database will raise a constraint error.
// This makes them your lowest-level, hardest-to-bypass data integrity guarantee.


// ── UserRole ──────────────────────────────────────────────────────────────────
// Why not four enums for the four personas?
//
// The PRD describes four user PERSONAS: Student Buyer, Normal Buyer,
// Student Seller, Reseller. But these personas are the INTERSECTION of two
// orthogonal properties: (1) what the user CAN DO (role) and (2) whether they
// are a verified student (verificationStatus).
//
// Modeling them as four enum values would mean:
//   - You'd have to update the role every time a user gets verified
//   - You'd need logic like "is STUDENT_BUYER or STUDENT_SELLER?" instead of
//     just checking verificationStatus === 'VERIFIED'
//   - Adding a 5th persona later (e.g., "Admin Seller") means a migration
//
// Two-field model:
//   role = BUYER + verificationStatus = VERIFIED  → "Student Buyer" persona
//   role = BUYER + verificationStatus = UNVERIFIED → "Normal Buyer" persona
//   role = SELLER + verificationStatus = VERIFIED  → "Student Seller" (5% fee)
//   role = SELLER + verificationStatus = UNVERIFIED → "Reseller" (8% fee)
//   role = BOTH  → User is currently both buying and selling
//
// This is cleaner, more flexible, and maps naturally to fee calculations.
enum UserRole {
  BUYER   // Can only purchase, cannot create a store
  SELLER  // Has a store, can list and sell items
  BOTH    // Active as both buyer and seller simultaneously
  ADMIN   // Platform staff: can moderate, resolve disputes, manage users
}

// ── VerificationStatus ────────────────────────────────────────────────────────
// This is what distinguishes a "Student Seller" from a "Reseller".
// UNVERIFIED is the default for ALL new users, including buyers.
// A verified student seller pays 5% commission; unverified (Reseller) pays 8%.
enum VerificationStatus {
  UNVERIFIED  // Default state. No special privileges or discounts.
  PENDING     // Student uploaded their ID; admin has not yet reviewed it.
  VERIFIED    // Confirmed student. Gets student fee rate and trust badge.
  REJECTED    // Admin reviewed the ID and determined it was invalid.
              // User should be notified with a reason and given a chance to resubmit.
}

// ── ListingStatus ─────────────────────────────────────────────────────────────
// Controls visibility and purchasability of a listing.
enum ListingStatus {
  DRAFT   // Seller is still writing the listing. Hidden from all buyers.
  ACTIVE  // Live and purchasable. Visible in search and store page.
  PAUSED  // Seller temporarily took it down (e.g., item not available right now).
  SOLD    // Stock = 0 after a purchase. Auto-set when stock hits zero.
}

// ── ConditionGrade ────────────────────────────────────────────────────────────
// 6-tier system from the PRD. These MUST match the illustrated condition guide
// shown on the listing creation form. Vague condition grading is the #1 source
// of disputes on second-hand tech marketplaces. Make each tier's criteria
// crystal clear with example photos in the UI.
enum ConditionGrade {
  NEW             // Sealed box, never opened. Original packaging intact.
  LIKE_NEW        // Opened but used 0-3 times. No marks, all accessories present.
  USED_EXCELLENT  // Light use. Barely visible wear. All functions perfect. Battery > 90%.
  USED_GOOD       // Normal use. Minor scratches under direct light. Battery > 80%.
  USED_FAIR       // Noticeable wear. Functional. Cosmetic damage. Battery 60-80%.
  FOR_PARTS       // Not fully functional. Sold for components/repair only.
}

// ── OrderStatus ───────────────────────────────────────────────────────────────
// This is the lifecycle of an order from creation to completion.
// Think of it as a state machine — each status can only transition
// to certain other statuses. Never jump states arbitrarily.
//
// Valid transitions:
//   PENDING_PAYMENT → PAYMENT_RECEIVED (via Paystack charge.success webhook)
//   PAYMENT_RECEIVED → PROCESSING      (seller acknowledges the order)
//   PROCESSING → IN_TRANSIT            (seller marks as shipped / meetup arranged)
//   IN_TRANSIT → DELIVERED             (buyer clicks Confirm Delivery, OR 7-day auto-release)
//   DELIVERED → COMPLETED              (escrow released to seller's wallet)
//   Any status → DISPUTED              (buyer opens a dispute — freezes everything)
//   DISPUTED → COMPLETED or REFUNDED   (admin resolves dispute)
//   PENDING_PAYMENT → CANCELLED        (buyer cancels before paying, or payment timed out)
enum OrderStatus {
  PENDING_PAYMENT   // Order record created, Paystack link generated. Not paid yet.
  PAYMENT_RECEIVED  // Paystack confirmed payment. Escrow record created. Seller notified.
  PROCESSING        // Seller acknowledged the order in their dashboard.
  IN_TRANSIT        // Seller marked shipped / meetup scheduled.
  DELIVERED         // Buyer confirmed receipt. Escrow release triggered.
  COMPLETED         // Escrow released to seller wallet. Review request sent.
  DISPUTED          // Buyer opened a formal dispute. Escrow frozen until resolved.
  CANCELLED         // Order cancelled. If paid, refund issued.
  REFUNDED          // Buyer received full refund. Inventory restored.
}

// ── EscrowStatus ─────────────────────────────────────────────────────────────
// The financial state of the funds held on behalf of a specific order.
enum EscrowStatus {
  HELD      // Funds sitting in U-Shop's Paystack balance. Seller cannot access yet.
  RELEASED  // Transfer to seller initiated. Wallet credited. Money on its way.
  REFUNDED  // Buyer was refunded. Seller receives nothing for this order.
}

// ── PayoutStatus ─────────────────────────────────────────────────────────────
// Tracks an individual withdrawal from a seller's wallet to their MoMo/bank.
enum PayoutStatus {
  PENDING     // Seller requested payout. Wallet debited. Waiting to call Paystack.
  PROCESSING  // Paystack transfer initiated. transfer_code stored on record.
  COMPLETED   // Paystack fired transfer.success webhook. Money arrived.
  FAILED      // Paystack fired transfer.failed. Wallet RE-CREDITED. Seller notified.
  REVERSED    // Bank reversed a completed transfer. Wallet RE-DEBITED. Rare but must handle.
}

// ── DisputeStatus ─────────────────────────────────────────────────────────────
enum DisputeStatus {
  OPEN            // Buyer just filed. Admin not yet involved.
  UNDER_REVIEW    // Admin is investigating. SLA: 72h first response (per PRD).
  RESOLVED_BUYER  // Admin ruled in buyer's favor. Escrow refunded to buyer.
  RESOLVED_SELLER // Admin ruled in seller's favor. Escrow released to seller.
  CLOSED          // Buyer withdrew the dispute. No action taken.
}

// ── MessageSenderRole ────────────────────────────────────────────────────────
enum MessageSenderRole {
  BUYER   // Message sent by the buyer in this transaction
  SELLER  // Message sent by the seller in this transaction
  SYSTEM  // Automated message: order status updates, reminders
}

2.2 The User Model

The User model is the center of gravity for the entire schema. It is the one entity that almost every other table references. Its design — specifically the two-field approach to roles — is the most important architectural decision in this schema.

How the 4 PRD personas map to the schema's 2-field model

Student Buyer

role = BUYER
verificationStatus = VERIFIED

Can browse and purchase. Gets a "Verified Student" badge on messages. Pays 0 commission.

Normal Buyer

role = BUYER
verificationStatus = UNVERIFIED

Can browse and purchase. No badge. Pays 0 commission.

Student Seller

role = SELLER (or BOTH)
verificationStatus = VERIFIED

Has a storefront. Platform fee: 5%. Shows "Verified Student Seller" badge on listings.

Reseller

role = SELLER (or BOTH)
verificationStatus = UNVERIFIED

Has a storefront. Platform fee: 8%. No badge. Can apply for verification.

model User {
    // ── Primary Key ──────────────────────────────────────────────────────────
  // We use CUID (Collision-Resistant Unique Identifiers) instead of auto-
  // incrementing integers. Why? Two reasons:
  //   1. Sequential integers are ENUMERABLE. An attacker can discover all
  //      user IDs by iterating /api/users/1, /api/users/2, /api/users/3...
  //   2. CUIDs are safe to expose in public URLs and API responses.
  // Prisma's @default(cuid()) generates a new CUID on every insert automatically.
  id                    String             @id @default(cuid())

    // ── Supabase Auth Link ────────────────────────────────────────────────────
  // supabaseId stores the UUID that Supabase Auth assigns to this user when
  // they sign up. It is the BRIDGE between two separate systems:
  //
  //   Supabase Auth (auth.users table) ──[supabaseId]──► Our users table
  //
  // When a user signs in, Supabase gives us a JWT token. That token contains
  // the user's Supabase UUID (auth.uid()). We look up supabaseId in OUR users
  // table to get our rich user record (role, verificationStatus, storeId, etc.).
  //
  // @unique: Two users can NEVER share the same Supabase account.
  // This column MUST match exactly what Supabase calls auth.users.id.
  supabaseId            String             @unique

  email                 String             @unique
  name                  String?            // Optional until user completes profile
  avatarUrl             String?            // URL to avatar in Supabase Storage
  phone                 String?            // For MoMo notification SMS

    // ── Role & Verification ───────────────────────────────────────────────────
  // See the role matrix table above for how these two fields combine to
  // produce the 4 user personas described in the PRD.
  // @default(BUYER): All new users start as buyers. They can upgrade to SELLER
  // by creating a store. ADMIN is only set manually by existing admins.
  role                  UserRole           @default(BUYER)
  verificationStatus    VerificationStatus @default(UNVERIFIED)

    // ── Student Verification Fields ───────────────────────────────────────────
  // These fields track the verification evidence and its outcome.
  // studentIdImagePath: The storage PATH (not full URL) of the uploaded student
  //   ID photo. We store paths so we can generate time-limited signed URLs when
  //   admins need to view them. After approval, schedule deletion per Ghana DPA.
  // studentEmail: The .edu.gh email they used (if auto-verified by domain).
  // verifiedAt/verifiedBy: Audit trail — who approved it and when.
  //   If verifiedBy = 'system:auto-domain', it was automatically approved.
  //   If verifiedBy = a user ID string, a human admin approved it.
  studentIdImagePath    String?
  studentEmail          String?
  verifiedAt            DateTime?
  verifiedBy            String?            // Admin userId or 'system:auto-domain'

    // ── Paystack Recipient Code ───────────────────────────────────────────────
  // This is the most important field for seller payouts.
  //
  // Background: Paystack's Transfer API doesn't transfer money to an "account
  // number" directly. You must first call POST /transferrecipient to register
  // a recipient (a MoMo number or bank account). Paystack validates it and
  // returns a "recipient_code" (e.g., "RCP_1a2b3c4d5e6f7g8h").
  //
  // You then use this recipient_code in POST /transfer to send money.
  // The recipient_code never expires — store it here so you only register
  // each seller's account ONCE, not on every payout.
  //
  // This field is NULL for buyers and unverified sellers who haven't added
  // their payout details yet. A seller cannot withdraw funds without it.
  paystackRecipientCode String?            // e.g., "RCP_1a2b3c..." — set during seller onboarding
  paystackRecipientType String?            // "mobile_money" or "ghipss" (bank)

    // ── Timestamps ───────────────────────────────────────────────────────────
  // Always include these. You WILL use them for debugging, analytics, and
  // compliance (Ghana Data Protection Act requires knowing when data was collected).
  // @updatedAt is maintained automatically by Prisma — no manual updates needed.
  createdAt             DateTime           @default(now())
  updatedAt             DateTime           @updatedAt

    // ── Relations (Virtual — not stored in the database) ─────────────────────
  // These fields are NOT columns. They're Prisma "relation fields" — Prisma
  // uses them to know how to JOIN tables when you use `include: { store: true }`.
  // The actual foreign keys live on the OTHER models (e.g., Store has owner_id).
  store                 Store?             // A user owns 0 or 1 store
  wallet                Wallet?            // A user has 0 or 1 wallet
  ordersAsBuyer         Order[]            // Orders placed by this user as buyer
  sentMessages          Message[]          // All messages this user has sent
  reviewsWritten        Review[]           // Reviews authored by this user
  disputesOpened        Dispute[]          // Disputes filed by this user as buyer

    // @@map: Tells Prisma the actual PostgreSQL table is named "users" (lowercase).
  // Convention: plural snake_case for table names.
  @@map("users")
}

2.3 The Store Model & Policy System

The Store model is where custom branding and seller policy configuration live. The most architecturally significant decision here is how the return/warranty policy is stored — as structured JSON rather than free text — so it remains queryable and enforceable.

model Store {
  id                String    @id @default(cuid())

    // ── Custom Store Handle (URL Slug) ────────────────────────────────────────
  // The handle is the unique text identifier for the store's public URL.
  // "janes-tech" → accessible at /store/janes-tech
  //
  // @unique creates a database-level uniqueness constraint.
  // This means: even if two concurrent requests both try to register
  // "janes-tech" at the exact same millisecond, PostgreSQL will reject the
  // second one with a unique constraint violation error. The database is the
  // final arbiter of uniqueness — application-level checks alone are NOT safe.
  //
  // Enforced rules (at application layer — validated before INSERT):
  //   - Lowercase only: stored and compared as lowercase always
  //   - 3–24 characters
  //   - Only alphanumeric + hyphens: /^[a-z0-9-]+$/
  //   - Cannot start or end with a hyphen
  //   - Not on the reserved words list (~200 slugs: "admin", "api", "search"...)
  handle            String    @unique

  name              String
  bio               String?   // Max 280 chars — enforced at app layer, not DB (flexibility)
  logoPath          String?   // Storage PATH: "stores/{storeId}/logo.webp" — NOT a full URL
  bannerPath        String?   // Storage PATH: "stores/{storeId}/banner.webp"

  <h2 id="escrow-model">2.6 The EscrowLedger Model — Financial Source of Truth</h2>
  <p>This is the most financially critical model in the schema. It records every Paystack reference ID and is the canonical record of where the money is at every point in the transaction lifecycle.</p>

  <div class="callout callout-paystack">
    <strong>Paystack "Collect and Transfer" Architecture</strong>
    <p>U-Shop collects 100% of the buyer's payment into the U-Shop Paystack account (charge). Funds sit there during the escrow period. On release, we call Paystack's Transfer API to send the seller's cut (<code>sellerAmount</code>) to their registered MoMo or bank account using their <code>recipientCode</code>. The EscrowLedger stores references to BOTH the incoming charge AND the outgoing transfer, creating a complete financial audit trail in a single database row.</p>
  </div>

<pre>
<span class="kw">model</span> EscrowLedger {
  id                   String       <span class="kw">@id @default</span>(cuid())

  <span class="comment-block">  // ── Amount Fields ─────────────────────────────────────────────────────────
  // heldAmount: The amount being held in escrow for the SELLER.
  //   = order.sellerAmount (total minus platform fee)
  //   This is what will be transferred to the seller on release.
  //
  // platformFee: U-Shop's revenue for this transaction.
  //   = order.platformFee
  //   This stays in U-Shop's Paystack account permanently.
  //   It is NOT transferred anywhere — it is U-Shop's earned revenue.
  //
  // Both stored here for convenience and audit clarity.
  // They should always sum to order.totalAmount.</span>
  heldAmount           Decimal      <span class="kw">@db.Decimal</span>(10, 2)
  platformFee          Decimal      <span class="kw">@db.Decimal</span>(10, 2)

  <span class="comment-block">  // ── Paystack Reference IDs ────────────────────────────────────────────────
  // These are the two most important external reference strings.
  //
  // paystackChargeRef: The Paystack transaction reference from the original
  //   payment. This is the same as order.paystackRef. We store it here again
  //   for quick lookups: "For this escrow record, what was the original charge?"
  //   You'll need this if you ever need to investigate a charge in the
  //   Paystack dashboard or via their API.
  //
  // paystackTransferRef: The reference WE generate when calling POST /transfer
  //   to pay the seller. We create this reference in our backend before calling
  //   Paystack so we can find the related Payout/EscrowLedger record when
  //   Paystack fires back the transfer.success or transfer.failed webhook.
  //   Format: "ushop_payout_{orderId}_{timestamp}"
  //   NULL until escrow is released (transfer hasn't happened yet).</span>
  paystackChargeRef    String                                <span class="comment-block">// Always present after payment</span>
  paystackTransferRef  String?                               <span class="comment-block">// Set when release is triggered</span>

  <span class="comment-block">  // ── Seller Recipient Code ─────────────────────────────────────────────────
  // A SNAPSHOT of the seller's Paystack recipient code at the time escrow was

  //
  // Why snapshot it? If a seller later CHANGES their payout account (e.g.,

The <code>available_balance</code> is a running total that you update on every credit/debit — think of it as a cached sum that must always equal the sum of all WalletTransaction amounts.</p>

  //   +850.00 → "Escrow released: Order #abc123"
  //
  // NEGATIVE = debit (money going OUT of the wallet):

  // The wallet is debited BEFORE calling Paystack (not after),
  // so the balance is immediately "locked" even before the transfer completes.
  // If the transfer fails, we RE-CREDIT the wallet in the transfer.failed handler.</span>
  amount               Decimal      <span class="kw">@db.Decimal</span>(10, 2)

  status               PayoutStatus <span class="kw">@default</span>(PENDING)

  <span class="comment-block">  // recipientType: "mobile_money" or "ghipss" (Ghana Interbank Payment & Settlement)
  // recipientDetails: JSON snapshot of the payout destination at time of request.
  //   { "network": "MTN", "accountNumber": "0551234567", "accountName": "Jane Doe" }
  //   Snapshot because the seller might change their details later.</span>
  recipientType        String
  recipientDetails     Json

  <span class="comment-block">  // paystackTransferCode: Returned by Paystack's POST /transfer endpoint.
  // Example: "TRF_abcdefghijklmno"
  // Store it immediately after calling Paystack. You'll need it to:

  <span class="kw">static</span> validateHandleFormat(handle: <span class="kw">string</span>): { valid: <span class="kw">boolean</span>; reason?: <span class="kw">string</span> } {
    <span class="kw">const</span> normalized = handle.toLowerCase().trim();

    <span class="kw">if</span> (normalized.length < <span class="num">3</span> || normalized.length > <span class="num">24</span>)

    <span class="kw">return</span> { valid: <span class="kw">true</span> };

      // request slipped through between our check and this INSERT.</span>

        <span class="kw">const</span> store = <span class="kw">await</span> tx.store.create({ data: { handle, ...data, ownerId } });

      }

    }
  }

  <h2>4.2 Row Level Security (RLS) Policies</h2>
  <p>RLS policies are rules written in SQL that PostgreSQL enforces on every single query. <strong>The Supabase ANON key (used in the frontend) is subject to RLS. The SERVICE_ROLE key (used only in Express) bypasses RLS entirely.</strong> This means your Express API can do anything — you are responsible for enforcing business rules there. RLS is your backup protection for the frontend.</p>

  <div class="callout callout-red">
    <strong>Critical: Enable RLS before writing policies</strong>
    <p>By default, PostgreSQL tables have RLS disabled — ALL rows are accessible to anyone with table access. Once you enable RLS on a table, ALL rows are DENIED until you add explicit allow policies. Enable RLS on every table before writing policies.</p>

  <h1>Indexing Strategy</h1>
  <p>An index is the difference between a query that completes in 2ms and one that takes 4 seconds and brings down your server. The indexes in the Prisma schema (Section 2) cover most cases, but the full-text search index and trigger must be applied manually via raw SQL in the Supabase SQL editor.</p>

  <div class="callout callout-blue">
    <strong>How a database index works (plain English)</strong>
    <p>Imagine your <code>listings</code> table has 100,000 rows. A query like <code>WHERE store_id = 'abc'</code> without an index makes PostgreSQL read all 100,000 rows to find the matching ones. This is called a "full table scan." An index on <code>store_id</code> is like the index in the back of a book — instead of reading every page, you jump straight to the entries for 'abc'. The cost: a small amount of extra storage and slightly slower INSERT/UPDATE operations. The payoff: queries go from seconds to milliseconds.</p>

  <h2>5.2 Manual SQL for Full-Text Search (Run Once in Supabase SQL Editor)</h2>
  <p>The tsvector-based search is the most performance-critical feature. The GIN index and trigger below cannot be expressed in Prisma's schema syntax — they must be applied manually after your first migration runs.</p>

<pre>
<span class="comment-block">-- ── Run in Supabase SQL Editor after first migration ─────────────────────────
-- This entire block should be idempotent (safe to re-run using IF NOT EXISTS).</span>

<span class="comment-block">-- Step 1: Add the search_vector column (if not added via migration already)
-- Prisma uses Unsupported("tsvector") which creates the column, but
-- double-check it exists before creating the trigger.</span>
<span class="kw">ALTER TABLE</span> listings <span class="kw">ADD COLUMN IF NOT EXISTS</span> search_vector tsvector;

<span class="comment-block">-- Step 2: Create the GIN index.
-- GIN (Generalized Inverted Index) is specifically designed for tsvector.
-- A regular BTree index on a tsvector column would be useless.
-- This single index makes full-text search instant at any table size.</span>
<span class="kw">CREATE INDEX IF NOT EXISTS</span> idx_listings_fts
  <span class="kw">ON</span> listings <span class="kw">USING GIN</span>(search_vector);

<span class="comment-block">-- Step 3: Create the trigger function.
-- to_tsvector('english', ...) tokenizes and stems English text.
--   "MacBooks" becomes the lexeme "macbook"
--   "running" becomes "run"
-- setweight assigns importance:
--   Weight 'A' (title) ranks higher in results than Weight 'B' (description)
-- coalesce(field, '') prevents null values from crashing the function.
-- The || operator concatenates the two tsvectors into one.</span>
<span class="kw">CREATE OR REPLACE FUNCTION</span> update_listing_search_vector()
<span class="kw">RETURNS TRIGGER AS</span> $$
<span class="kw">BEGIN</span>
  NEW.search_vector :=
    setweight(to_tsvector(<span class="str">'english'</span>, coalesce(NEW.title, <span class="str">''</span>)), <span class="str">'A'</span>) ||
    setweight(to_tsvector(<span class="str">'english'</span>, coalesce(NEW.description, <span class="str">''</span>)), <span class="str">'B'</span>);
  <span class="kw">RETURN NEW</span>;
<span class="kw">END</span>;
$$ <span class="kw">LANGUAGE plpgsql</span>;

<span class="comment-block">-- Step 4: Attach the trigger.
-- BEFORE INSERT OR UPDATE: runs the function BEFORE writing to disk.
-- FOR EACH ROW: runs once per affected row (not once per statement).
-- The function updates NEW.search_vector, which is then written to disk.</span>
<span class="kw">DROP TRIGGER IF EXISTS</span> listings_search_vector_update <span class="kw">ON</span> listings;
<span class="kw">CREATE TRIGGER</span> listings_search_vector_update
  <span class="kw">BEFORE INSERT OR UPDATE ON</span> listings
  <span class="kw">FOR EACH ROW EXECUTE FUNCTION</span> update_listing_search_vector();

<span class="comment-block">-- Step 5: Backfill existing rows (run once).
-- New rows are covered by the trigger. Old rows need a one-time backfill.
-- This statement may take a few seconds if you already have many listings.</span>
<span class="kw">UPDATE</span> listings <span class="kw">SET</span> search_vector = search_vector; <span class="comment-block">-- Triggers the trigger on every row</span>

<span class="comment-block">-- Step 6: Partial indexes for high-frequency filtered queries.
-- A partial index only indexes rows matching the WHERE condition.
-- This makes the index smaller and faster — and PostgreSQL will use it
-- automatically when your query includes the same WHERE condition.

-- Only index ACTIVE listings (DRAFT/PAUSED/SOLD are rarely searched)</span>
<span class="kw">CREATE INDEX IF NOT EXISTS</span> idx_listings_active_store
  <span class="kw">ON</span> listings(store_id, created_at <span class="kw">DESC</span>)
  <span class="kw">WHERE</span> status = <span class="str">'ACTIVE'</span>;

<span class="kw">CREATE INDEX IF NOT EXISTS</span> idx_listings_active_category_price
  <span class="kw">ON</span> listings(category_id, price)
  <span class="kw">WHERE</span> status = <span class="str">'ACTIVE'</span>;

<span class="comment-block">-- Only index orders pending escrow release (the 7-day cron uses this)</span>
<span class="kw">CREATE INDEX IF NOT EXISTS</span> idx_orders_pending_escrow
  <span class="kw">ON</span> orders(escrow_release_at)
  <span class="kw">WHERE</span> status = <span class="str">'PAYMENT_RECEIVED'</span>;

<span class="comment-block">-- Usage example — search query in Prisma service:
-- const results = await prisma.$queryRaw`
--   SELECT id, title, price, condition, store_id,
--          ts_rank(search_vector, query) AS rank
--   FROM listings, plainto_tsquery('english', ${searchTerm}) query
--   WHERE search_vector @@ query
--     AND status = 'ACTIVE'
--   ORDER BY rank DESC
--   LIMIT ${limit} OFFSET ${offset}
-- `;</span>
</pre>

  <div class="insight">
    <div class="ins-l">Performance Principle — A Final Word</div>
    <p>Indexes make reads faster but writes slightly slower (PostgreSQL must update every index on INSERT/UPDATE). For U-Shop, this trade-off is overwhelmingly positive: reads (browsing, searching) happen 50–100× more often than writes (listing creation, order placement). <strong>The only tables where you should be cautious about over-indexing are <code>wallet_transactions</code> and <code>webhook_events</code></strong>, which receive very frequent INSERTs. Keep only the indexes listed above on those tables.</p>
