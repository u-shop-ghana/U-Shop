1.1 Why Two Separate Apps (Next.js + Express)?

This is the single most important architectural decision. Here is the honest reasoning:

Express handles what Next.js cannot

  • Webhooks need to be always-on. Paystack expects a response within 20 seconds. Vercel serverless functions have cold starts and execution limits — they can fail silently. An always-running Express container on Render never cold-starts.
  • Background jobs need persistence. The escrow auto-release cron (runs every minute), the 7-day reminder sequence, the reconciliation job — none of these can live in a serverless function. They need a persistent process and a Redis-backed job queue (BullMQ).
  • Business logic needs isolation. All financial calculations — fee splits, escrow state machines, payout ledger — belong in one testable, auditable Express service layer, not scattered across Next.js API routes.

Next.js handles what Express should not

  • Server-side rendering for SEO. Store pages at /store/janes-tech must be rendered as HTML so Google indexes them. Express returns JSON — it doesn't render React.
  • Supabase Auth session management. The @supabase/ssr library integrates directly with Next.js cookies and middleware. Keep auth token handling in Next.js where Supabase designed it to live.
  • Edge-layer routing. Custom store URL validation, auth redirects, and 301 redirects for changed handles all run on Vercel's Edge network — zero latency, no server involved.

1.2 The Full Request Journey: Store Visit → Paystack Checkout

Phase A — Visiting the Store Page

1

Browser → Vercel Edge Network

Browser sends GET /store/janes-tech. Before Next.js runs, the request hits Vercel's global CDN. If a cached HTML version of this store exists (TTL: 60s), it's returned from the nearest edge node instantly. Cache miss → route to Next.js server function.

2

Next.js Middleware intercepts — zero database queries

The middleware.ts file at the Edge validates the store handle format with a regex (/^[a-z0-9][a-z0-9-]{1,22}[a-z0-9]$/) and checks the user's Supabase session cookie. If the handle is malformed → immediate 404 redirect. No database query wasted.

3

Next.js Server Component fetches store data from Express

The app/store/[handle]/page.tsx Server Component runs on the server and calls GET /api/v1/stores/janes-tech on Express, including the auth token if the user is logged in. Express queries Supabase's PostgreSQL via Prisma, returns store + active listings. The browser receives fully rendered HTML — not a loading spinner.

4

HTML served + React hydrates interactive elements

Vercel caches the rendered HTML with next: { revalidate: 60 }. React hydrates the cart button, image gallery, and message thread client-side. The page is interactive in milliseconds — all layout content was already in the HTML.

Phase B — "Buy Now" → Paystack Checkout → Escrow

5

Client Component → Express: Initiate Order

"Buy Now" is a Client Component. On click, it makes an authenticated POST /api/v1/orders/initiate to Express with the listing ID, quantity, and shipping method. The Supabase JWT travels in the Authorization: Bearer <token> header.

6

Express: Validate, snapshot policy, create Order, call Paystack

Express's OrderService: (a) verifies the JWT via Supabase, (b) checks listing is Active and in stock, (c) calculates the platform fee (5% for verified students, 8% for resellers), (d) snapshots the seller's return policy into the Order record — this snapshot is immutable after creation, (e) decrements stock in a database transaction, (f) calls Paystack's POST /transaction/initialize API to get a checkout URL, (g) returns the Paystack authorization_url and the internal order ID to the frontend.

7

Frontend redirects to Paystack's hosted checkout page

The user is redirected to Paystack's authorization_url — a page hosted entirely on Paystack's servers. The user selects payment method: card, MTN MoMo, Vodafone Cash, or AirtelTigo. U-Shop never sees card details or MoMo PINs. Paystack handles PCI compliance entirely.

8

Payment processed → Paystack redirects back + fires webhook

After the user pays, Paystack does two things simultaneously: (a) redirects the user's browser to your callback_url (e.g., /checkout/complete?reference=abc123), and (b) fires a charge.success webhook to your Express server. Never trust the redirect alone — a bad actor can manually visit the callback URL. Only trust the webhook.

9

Express webhook handler: verify HMAC → update Order → create Escrow

The webhook handler verifies the HMAC-SHA512 signature using your Paystack secret key, looks up the Order by the Paystack reference, updates status to PAYMENT_RECEIVED, creates an Escrow record (HOLDING), schedules reminder jobs in Redis (Day 3/5/6/6.5), and sends an email to the seller. All of this runs inside a Prisma database transaction — if any step fails, all steps roll back.

10

Callback route: Next.js verifies and shows confirmation

The /checkout/complete page calls GET /api/v1/orders/[id] on Express to get the latest order status. If PAYMENT_RECEIVED, show the success screen. If still PENDING_PAYMENT (webhook hasn't fired yet), poll every 2 seconds for up to 30 seconds. This handles the race between the redirect and the webhook.

Paystack Initialize vs. Inline vs. Popup — Why Initialize

Paystack offers three integration modes. Use Initialize (server-side redirect) for U-Shop, not the inline embed or popup. The Initialize API call happens on your Express server using your secret key — never exposed to the browser. The popup and inline modes require your public key in the browser, which is less secure for server-side order creation logic. Initialize is also more reliable on slow mobile connections, which is the dominant device type for your campus audience.

02 — Frontend Architecture

Next.js 14 Frontend Design

2.1 Rendering Strategy: Server vs. Client Components

In Next.js 14's App Router, every component is a React Server Component (RSC) by default — it runs on the server, can fetch data directly, and never ships its code to the browser. You opt into client-side behaviour by adding 'use client'. The rule: a component needs 'use client' only if it uses hooks (useState, useEffect), browser APIs, or event handlers. Everything else stays server-side.

Server Component (RSC) — Default

  • Store page /store/[handle] — fetches store + listings, renders SEO HTML
  • Listing detail page — product info, seller details, policy accordion
  • Search results page — runs query, renders filtered grid
  • Category pages — fetches listings, no interaction needed at load time
  • Homepage feed — featured + recently added listings
  • Order history page — pure data display
  • Dashboard layout shell — wraps authenticated pages, fetches user once
  • Admin panel pages — heavy data, no real-time updates needed

Client Component ('use client')

  • Cart button + cart drawer — maintains cart state via useState
  • Image gallery / lightbox — carousel needs browser pointer events
  • Checkout redirect button — calls Express to get Paystack URL, then window.location.href
  • In-app message chat — Supabase Realtime subscription requires WebSocket
  • Search bar with autocomplete — debounced onChange
  • Store policy builder form — complex multi-field controlled state
  • Listing creation form — multi-step with file upload previews
  • Meetup confirmation code widget — 6-digit input with timer

Server Actions

  • Confirm delivery — buyer marks received, triggers escrow release flow
  • Leave review — post-transaction, calls Express mutation
  • Open dispute — uploads evidence to Supabase Storage, calls Express
  • Update profile — simple user data mutations
  • Save / unsave listing — toggle wishlist state

Middleware (Edge)

  • Auth guard — redirects unauthenticated users from /dashboard
  • Role guard — blocks non-admins from /admin
  • Store handle validation — regex check, no DB query
  • 301 redirect for changed store handles — checks previousHandle
  • Bot / crawler detection — lightweight check at edge layer

2.2 Middleware: Dynamic Store URL Routing

// apps/web/middleware.ts — Runs at Vercel Edge BEFORE any page renders.
// Must be fast. No database queries. No heavy imports. Pure logic only.
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';
import { createServerClient } from '@supabase/ssr';

// Valid handle: starts + ends with alphanumeric, hyphens allowed in middle,
// total length 3–24 characters. Rejects path traversal and special chars.
const VALID_HANDLE = /^[a-z0-9][a-z0-9-]{1,22}[a-z0-9]$/;
const PROTECTED = ['/dashboard', '/checkout', '/admin'];

export async function middleware(req: NextRequest) {
  const { pathname } = req.nextUrl;

  // 1. Validate store handle format — zero DB queries needed
  if (pathname.startsWith('/store/')) {
    const handle = pathname.split('/')[2];
    if (!VALID_HANDLE.test(handle))
      return NextResponse.redirect(new URL('/not-found', req.url));
  }

  // 2. Refresh Supabase session from cookies (fast — just reads a cookie)
  let res = NextResponse.next({ request: req });
  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    { cookies: { get: (n) => req.cookies.get(n)?.value,
                  set: (n,v,o) => res.cookies.set(n,v,o),
                  remove: (n,o) => res.cookies.set(n,'',o) } }
  );
  const { data: { session } } = await supabase.auth.getSession();

  // 3. Guard protected routes
  const isProtected = PROTECTED.some(p => pathname.startsWith(p));
  if (isProtected && !session) {
    const url = new URL('/auth/login', req.url);
    url.searchParams.set('redirectTo', pathname);
    return NextResponse.redirect(url);
  }

  return res;
}

export const config = { matcher: [
  '/((?!_next/static|_next/image|favicon.ico|.*\\.(?:png|jpg|svg|webp)$).*)'
]};
Store Handle Change → 301 Redirect

Sellers can change their handle once. The old handle is stored in store.previousHandle. The store's Server Component checks for this and calls Next.js's redirect() which issues a proper 301 permanent redirect. Vercel's CDN caches this redirect — old URLs keep working indefinitely and Google's index updates automatically within days.

3.2 The Critical Middleware Order in index.ts

The order in which middleware is registered in index.ts is not a stylistic choice — it is a functional requirement. Getting this order wrong will silently break webhook signature verification.

// apps/api/src/index.ts — middleware registration order matters

const app = express();

// ── Step 1: Security headers ─────────────────────────────────────────
app.use(helmet());

// ── Step 2: CORS ──────────────────────────────────────────────────────
app.use(cors({ origin: process.env.FRONTEND_URL, credentials: true }));

// ── Step 3: WEBHOOK ROUTES — registered BEFORE body parsing ──────────
// PAYSTACK SIGNATURE VERIFICATION REQUIRES THE RAW, UNPARSED REQUEST BODY.
// Once express.json() runs, the raw bytes are gone and HMAC verification fails.
// This is the #1 webhook integration mistake. Register webhooks FIRST.
app.use('/webhooks', webhookRoutes);

// ── Step 4: Body parsing (AFTER webhook routes) ───────────────────────
app.use(express.json({ limit: '10mb' }));
app.use(express.urlencoded({ extended: true }));

// ── Step 5: Logging ───────────────────────────────────────────────────
app.use(morgan(process.env.NODE_ENV === 'production' ? 'combined' : 'dev'));

// ── Step 6: Rate limiting ─────────────────────────────────────────────
app.use('/api', rateLimiter.general);

// ── Step 7: All other routes ──────────────────────────────────────────
app.use('/api/v1/auth', authRoutes);
app.use('/api/v1/stores', storeRoutes);
app.use('/api/v1/listings', listingRoutes);
app.use('/api/v1/orders', orderRoutes);
app.use('/api/v1/payments', paymentRoutes);
app.use('/api/v1/users', userRoutes);
app.use('/api/v1/admin', adminRoutes);

// ── Step 8: Error handler — MUST be last ──────────────────────────────
app.use(errorHandler);

3.3 Order Initiation — Full Service Implementation

// apps/api/src/services/order.service.ts
// This is the business logic layer. No req/res objects here.

async function initiateOrder(dto: {
  buyerId: string; listingId: string;
  quantity: number; shippingAddress: object;
}) {
  // 1. Fetch listing with store + seller in a single query
  const listing = await prisma.listing.findUniqueOrThrow({
    where: { id: dto.listingId, status: 'ACTIVE' },
    include: { store: { include: { owner: true } } }
  });

  // 2. Business rules
  if (listing.store.ownerId === dto.buyerId)
    throw new ForbiddenError('You cannot purchase your own listing');
  if (listing.stock < dto.quantity)
    throw new BadRequestError('Insufficient stock');

  // 3. Fee calculation based on seller verification status
  const isStudentSeller = listing.store.owner.verificationStatus === 'VERIFIED';
  const feeRate    = isStudentSeller ? 0.05 : 0.08;
  const totalAmount = listing.price.mul(dto.quantity);
  const platformFee = totalAmount.mul(feeRate);
  const sellerAmount = totalAmount.sub(platformFee);

  // 4. Generate a unique Paystack reference for this transaction
  // Format: ushop_[timestamp]_[random] — helps with debugging in Paystack dashboard
  const paystackRef = `ushop_${Date.now()}_${crypto.randomUUID().slice(0,8)}`;

  // 5. Database transaction — all-or-nothing. If Paystack call fails,
  // the DB is rolled back and no dangling order record is left behind.
  const order = await prisma.$transaction(async (tx) => {
    const newOrder = await tx.order.create({ data: {
      buyerId:         dto.buyerId,
      storeId:         listing.storeId,
      totalAmount,     platformFee,    sellerAmount,
      paystackRef,
      shippingAddress: dto.shippingAddress,
      // POLICY SNAPSHOT — immutable at point of purchase.
      // Even if the seller updates their policy tomorrow, this
      // order is governed by what they had at checkout time.
      policySnapshot:  listing.store.returnPolicy ?? {},
      items: { create: { listingId: dto.listingId, quantity: dto.quantity, unitPrice: listing.price } }
    }});
    // Reserve stock immediately to prevent overselling
    await tx.listing.update({
      where: { id: dto.listingId },
      data: { stock: { decrement: dto.quantity } }
    });
    return newOrder;
  });

  // 6. Initialize Paystack transaction (outside the DB transaction —
  // Paystack is an external service and cannot be rolled back with Prisma)
  const buyer = await prisma.user.findUniqueOrThrow({ where: { id: dto.buyerId } });
  const paymentData = await PaystackService.initializeTransaction({
    email:     buyer.email,
    amount:    toKobo(totalAmount),  // Paystack uses lowest denomination (kobo/pesewas)
    reference: paystackRef,
    callback_url: `${process.env.FRONTEND_URL}/checkout/complete?orderId=${order.id}`,
    metadata: {
      orderId:  order.id,
      storeId:  listing.storeId,
      sellerId: listing.store.ownerId,
    },
    // Restrict to channels available in Ghana
    channels: ['card', 'mobile_money', 'bank'],
  });

  return {
    orderId:          order.id,
    authorizationUrl: paymentData.authorization_url, // Frontend redirects here
    accessCode:       paymentData.access_code,
    reference:        paystackRef,
  };
}
04 — Data Layer

Supabase + Prisma Data Architecture

4.1 Connection Pooling — The Two-URL Pattern

This is the most common source of production outages for teams using Supabase + Prisma. PostgreSQL has a maximum connection limit. Supabase's free tier allows ~60 connections; Pro tier allows more. If 200 simultaneous requests each try to open a database connection, you hit the limit and queries fail with too many connections.

The Solution: PgBouncer (port 6543) for queries, Direct (port 5432) for migrations

Supabase provides a built-in connection pooler called PgBouncer. Your Express server connects through it (port 6543) — PgBouncer multiplexes thousands of app requests through ~20 real database connections. Prisma's migration CLI needs a direct connection (port 5432) because DDL operations don't work through PgBouncer's transaction-mode pooling. This is why you have two database URLs.

# apps/api/.env

# ── ALL application queries go through PgBouncer (port 6543) ─────────
# ?pgbouncer=true disables Prisma's prepared statements, which PgBouncer
# doesn't support in transaction mode.

# ── MIGRATIONS ONLY — bypasses PgBouncer directly (port 5432) ────────
# Only used when you run: prisma migrate dev / prisma migrate deploy
DIRECT_URL="postgresql://postgres.[ref]:[password]@[host]:5432/postgres"

4.3 Full-Text Search Setup (PostgreSQL tsvector)

-- Run in Supabase SQL Editor once after schema creation.
-- The GIN index makes search queries instant even at 100k+ listings.

-- GIN = Generalized Inverted Index. Specifically designed for full-text search.
CREATE INDEX idx_listings_search ON listings USING GIN(search_vector);

-- This trigger auto-populates the search_vector column whenever a listing
-- is created or updated. You never have to manually update it.
CREATE OR REPLACE FUNCTION update_listing_search_vector()
RETURNS TRIGGER AS $$
BEGIN
  -- 'A' weight = title (highest priority in search ranking)
  -- 'B' weight = description (lower priority)
  NEW.search_vector :=
    setweight(to_tsvector('english', coalesce(NEW.title, '')), 'A') ||
    setweight(to_tsvector('english', coalesce(NEW.description, '')), 'B');
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER listings_search_update
  BEFORE INSERT OR UPDATE ON listings
  FOR EACH ROW EXECUTE FUNCTION update_listing_search_vector();

-- SELECT * FROM listings
-- WHERE search_vector @@ plainto_tsquery('english', 'macbook pro m2')

05 — Paystack Webhooks & Payouts

Paystack: Complete Integration Architecture

5.1 How the Money Flows

Paystack does not natively split funds the way Stripe Connect does. Instead, you build a virtual escrow in your own database and use Paystack's Transfer API to pay out sellers after delivery is confirmed. Here is the complete money flow for a GH₵ 1,000 laptop sale:

Buyer pays via MTN MoMo or cardGH₵ 1,000.00
→Paystack processing fee (~1.95% + GH₵ 0.50 cap) deducted by Paystack− GH₵ 19.50
→Net amount received into U-Shop's Paystack accountGH₵ 980.50
[ ESCROW STATE: HOLDING — funds sit in U-Shop Paystack account ]
→Buyer confirms delivery (or 7-day auto-release fires)Escrow released
→U-Shop platform fee (8% of GH₵ 1,000) retained by platformGH₵ 80.00
→Seller payout via Paystack Transfer API to MoMo or bankGH₵ 900.50
U-Shop keeps: GH₵ 80.00 (platform fee) + Paystack fee is buyer-side cost
Escrow is a database concept, not a Paystack feature

Unlike Stripe Connect, Paystack doesn't have a built-in escrow mechanism. When a buyer pays, the full amount goes into your U-Shop Paystack account. Your Escrow table in the database tracks which funds are "held" for which order. The funds are physically in your Paystack account the whole time — your Escrow record is just a ledger entry saying "GH₵ 1,000 of the money in our account is reserved for order xyz until delivery is confirmed." When delivery is confirmed, you instruct Paystack to transfer the seller's share via the Transfer API. This means you must have enough float in your Paystack balance to process all pending payouts — factor this into cash flow planning.

5.2 Seller Payout Recipient Setup (One-Time)

Before you can transfer money to a seller, Paystack requires you to create a Transfer Recipient — a record of who to pay and how. This is done once when the seller sets up their payout method. Store the recipient_code on the User record.

// apps/api/src/lib/paystack.ts — Paystack API wrapper
const PAYSTACK_BASE = 'https://api.paystack.co';
const headers = {
  Authorization: `Bearer ${process.env.PAYSTACK_SECRET_KEY}`,
  'Content-Type': 'application/json',
};

// Create a Transfer Recipient for a seller (called once during seller onboarding)
export async function createTransferRecipient(data: {
  type: 'mobile_money' | 'ghipss';  // mobile_money = MoMo, ghipss = bank
  name: string;           // Seller's full name
  account_number: string; // MoMo number or bank account number
  bank_code: string;      // "MTN" for MTN MoMo, bank code for banks
  currency: 'GHS';
}) {
  const res = await fetch(`${PAYSTACK_BASE}/transferrecipient`, {
    method: 'POST', headers,
    body: JSON.stringify(data),
  });
  const json = await res.json();
  if (!json.status) throw new Error(`Paystack recipient creation failed: ${json.message}`);
  // Store json.data.recipient_code on the User record: user.paystackRecipientCode
  return json.data;
}

// Initialize a transaction — returns authorization_url to redirect the buyer to
export async function initializeTransaction(data: {
  email: string; amount: number; reference: string;
  callback_url: string; metadata: object; channels: string[];
}) {
  const res = await fetch(`${PAYSTACK_BASE}/transaction/initialize`, {
    method: 'POST', headers, body: JSON.stringify(data),
  });
  const json = await res.json();
  if (!json.status) throw new Error(`Paystack initialize failed: ${json.message}`);
  return json.data; // { authorization_url, access_code, reference }
}

// Initiate a transfer to a seller's recipient code
export async function initiateTransfer(data: {
  source: 'balance'; amount: number; recipient: string;
  reason: string; reference: string;
}) {
  const res = await fetch(`${PAYSTACK_BASE}/transfer`, {
    method: 'POST', headers, body: JSON.stringify(data),
  });

5.3 Escrow Release & Seller Payout — Complete Flow

1

Buyer clicks "Confirm Delivery" (or 7-day auto-release fires)

A Server Action calls PATCH /api/v1/orders/:id/confirm-delivery on Express.

2

Express: Update Order + Escrow in a DB transaction

Inside a Prisma transaction: Order status → COMPLETED, Escrow status → RELEASED, releasedAt timestamped. Credit the seller's Wallet (availableBalance += sellerAmount) and create a WalletTransaction ledger entry. All atomic — either all happens or none does.

3

Seller's wallet balance updated — payout is not instant

The seller's wallet now shows the credited amount. They can see it in their dashboard. The actual money transfer to their MoMo or bank happens when they explicitly request a payout, or automatically when their balance exceeds the auto-payout threshold (optional feature).

4

Seller requests payout → Express calls Paystack Transfer API

When the seller clicks "Withdraw GH₵ 900": Express validates the amount against their availableBalance, deducts the amount from the wallet in a DB transaction, creates a Payout record (PENDING), then calls Paystack's POST /transfer using the seller's stored paystackRecipientCode. Paystack returns a transfer_code — stored on the Payout record for tracking.

5

Paystack fires transfer.success or transfer.failed webhook

If success: Payout status → COMPLETED. Notify seller via email/in-app. If failed: Payout status → FAILED, store failureReason, reverse the wallet debit (add the amount back to availableBalance), notify seller with the failure reason and instructions to retry.

5.4 Webhook Handler — Secure HMAC Verification

Paystack signs every webhook with an HMAC-SHA512 hash of the request body using your secret key. Your server recomputes the hash and compares it. If they match, the webhook is genuine. This must happen before you act on any webhook data.

// apps/api/src/controllers/webhooks.controller.ts
import crypto from 'crypto';

export async function handlePaystack(req: Request, res: Response) {

  // ── Step 1: Verify HMAC-SHA512 signature ──────────────────────────
  // Paystack sends the hash in the 'x-paystack-signature' header.
  // We recompute it from the raw body using our secret key.
  // req.body is a Buffer here because this route uses express.raw()
  const signature = req.headers['x-paystack-signature'] as string;
  const expectedHash = crypto
    .createHmac('sha512', process.env.PAYSTACK_SECRET_KEY!)
    .update(req.body)   // req.body is the raw Buffer — must NOT be JSON-parsed yet
    .digest('hex');

  if (signature !== expectedHash) {
    // Signature mismatch — NOT from Paystack. Could be an attack.
    console.warn('Paystack webhook signature mismatch');
    return res.status(401).send('Unauthorized');
  }

  // ── Step 2: Parse the body now that we've verified it ─────────────
  const event = JSON.parse(req.body.toString());

  // ── Step 3: Idempotency — check if we've processed this event ────
  // Paystack retries webhooks if it doesn't receive a 200 within 20s.
  // The same event can arrive 2–5 times. Without this check, you'd
  // double-credit wallets, double-release escrows, double-send emails.
  const eventId = event.id?.toString() ?? event.data?.reference;
  const existing = await prisma.webhookEvent.findUnique({
    where: { externalId: eventId }
  });
  if (existing?.processed) {
    return res.status(200).json({ received: true, status: 'already_processed' });
  }

  // ── Step 4: Log the raw event BEFORE processing ───────────────────
  // If your server crashes mid-processing, you still have the raw event.
  await prisma.webhookEvent.upsert({
    where: { externalId: eventId },
    create: { externalId: eventId, source: 'paystack', eventType: event.event, payload: event },
    update: {},
  });

  // ── Step 5: Acknowledge Paystack IMMEDIATELY ──────────────────────
  // CRITICAL: Paystack marks the delivery failed if no 200 within 20 seconds.
  // Send 200 first, then process asynchronously. Never make Paystack wait
  // for your database writes before acknowledging receipt.
  res.status(200).json({ received: true });

  // ── Step 6: Process the event (after response is sent) ───────────
  try {
    switch (event.event) {
      case 'charge.success':
        await WebhookService.handleChargeSuccess(event.data);
        break;
      case 'transfer.success':
        await WebhookService.handleTransferSuccess(event.data);
        break;
      case 'transfer.failed':
        await WebhookService.handleTransferFailed(event.data);
        break;
      case 'transfer.reversed':
        await WebhookService.handleTransferReversed(event.data);
        break;
      default:
        console.log(`Unhandled Paystack event: ${event.event}`);
    }
    await prisma.webhookEvent.update({
      where: { externalId: eventId },
      data: { processed: true, processedAt: new Date() }
    });
  } catch (err) {
    // Processing failed after 200 was sent. Log it — the reconciliation
    // job will retry it. This order is NOT lost.
    await prisma.webhookEvent.update({
      where: { externalId: eventId },
      data: { error: String(err) }
    });
    console.error('Webhook processing error', err);
  }
}

5.6 Reconciliation: Catching Missed Webhooks

// apps/api/src/jobs/reconciliation.job.ts
// Runs every 5 minutes via BullMQ cron job.
// Problem: Your server was briefly down when Paystack fired a webhook.
// Paystack retried 3 times, all failed. Buyer paid, order stuck at PENDING_PAYMENT.
// This job catches those cases and heals them automatically.

async function reconcileStuckOrders() {
  // Orders stuck in PENDING_PAYMENT for more than 10 minutes.
  // If payment was processed, Paystack confirms in < 2 minutes normally.
  const stuckOrders = await prisma.order.findMany({
    where: { status: 'PENDING_PAYMENT',
             createdAt: { lte: new Date(Date.now() - 10 * 60 * 1000) } }
  });

  for (const order of stuckOrders) {
    try {
      // Ask Paystack directly: did this transaction succeed?
      const res = await fetch(`https://api.paystack.co/transaction/verify/${order.paystackRef}`,
        { headers });
      const { data } = await res.json();

      if (data.status === 'success') {
        // Payment went through — we just missed the webhook. Process it now.
        await WebhookService.handleChargeSuccess(data);
        console.log(`Reconciled order: ${order.id}`);
      } else if (data.status === 'failed') {
        // Payment genuinely failed — cancel the order and restock
        await prisma.$transaction([
          prisma.order.update({ where: { id: order.id }, data: { status: 'CANCELLED' }}),
          prisma.listing.update({ where: { id: order.items[0].listingId },
            data: { stock: { increment: order.items[0].quantity } }}
          ),
        ]);
      }
      // If status is 'pending' or 'abandoned', leave it — Paystack is still processing
    } catch (err) {
      console.error(`Reconciliation failed for ${order.id}:`, err);
    }
  }
}
06 — Security & Scaling

Security Architecture & Scaling Strategy

6.1 Auth Token Flow: Supabase JWT → Express

1

User logs in via Supabase — JWT stored in httpOnly cookie

The Supabase JS client handles login. The JWT is stored in an httpOnly secure cookie — JavaScript cannot read this cookie, which prevents XSS attacks from stealing auth tokens.

2

Server Components and Client Components extract the JWT differently

In Server Components: @supabase/ssr's createServerClient reads the cookie server-side and provides the session. In Client Components: supabase.auth.getSession() reads from browser memory. Both give you an access_token to pass to Express.

3

Express middleware verifies the JWT via Supabase Admin SDK

The authenticate middleware calls supabaseAdmin.auth.getUser(token). This verifies the JWT's HMAC signature against Supabase's keys and returns the Supabase User object. It then queries Prisma for the full application User record (findUnique({ where: { supabaseId } })) and attaches it to req.user.

4

Role-based authorization runs after authentication

requireRole('ADMIN') checks req.user.role. requireStoreOwner() checks that the authenticated user's store matches the requested store handle. These middleware functions run in the route definition chain after authenticate.

6.2 Rate Limiting (Redis-Backed Sliding Window)

Rate limiting uses Upstash Redis — not in-memory. Your Express container may run as multiple instances; an in-memory rate limiter only tracks its own instance. Redis provides a shared, consistent counter across all instances.

async function getStoreByHandle(handle: string) {
  const key = `store:${handle}`;
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);

  const store = await prisma.store.findUnique({
    where: { handle },
    include: { owner: { select: { verificationStatus: true, name: true } },
               listings: { where: { status: 'ACTIVE' } } }
  });
  if (!store) throw new NotFoundError(`Store not found: ${handle}`);

  await redis.set(key, JSON.stringify(store), { ex: 300 }); // 5-min TTL
  return store;
}

6.4 Production Security Checklist

  • [!]
    Use Decimal type for all money — never Float. 0.1 + 0.2 !== 0.3 in JavaScript floats. Every price, fee, escrow amount, and wallet balance must use Prisma's Decimal type (@db.Decimal(10, 2)). Getting this wrong corrupts ledgers silently.
  • [!]
    Webhook route before JSON parser — no exceptions. app.use('/webhooks', webhookRoutes) must appear before app.use(express.json()) in index.ts. HMAC-SHA512 verification requires the raw request body bytes. Once JSON parsing runs, those bytes are gone.
  • [✓]
    PAYSTACK_SECRET_KEY — Express backend only, never frontend. Never prefix it with NEXT_PUBLIC_. Anyone with your secret key can drain your Paystack balance via the Transfer API.
  • [✓]
    SUPABASE_SERVICE_ROLE_KEY — Express backend only. This key bypasses Row Level Security entirely. In the frontend environment, only NEXT_PUBLIC_SUPABASE_ANON_KEY should exist — it has limited permissions enforced by RLS.
  • [✓]
    Webhook idempotency — the WebhookEvent table with externalId unique constraint. Paystack retries webhooks. Without deduplication, a retry will double-credit a wallet or double-release an escrow. This is a financial integrity issue, not a bug.
  • [✓]
    CORS set to production domain only. origin: 'https://ushop.com' — not '*'. This prevents other websites from making authenticated calls to your API on behalf of your logged-in users (CSRF via CORS).
  • [✓]
    Payout: transfer_code stored on Payout record. When Paystack's transfer.failed webhook fires, you identify which Payout failed by matching data.transfer_code to the stored value. Without this you can't reconcile failed payouts.
  • [⚠]
    Ghana Data Protection Act — student ID deletion SLA. Student ID photos in the verification-docs bucket must be deleted within 30 days of a verification decision. Schedule a BullMQ job at the time of upload that fires at day 30 and deletes the file from Supabase Storage, then clears the studentIdImagePath field on the User record.
  • [⚠]
    Paystack balance float — plan for peak season. Since funds flow through your Paystack account (not a Stripe-style sub-account per seller), you need enough balance to pay out all pending escrows. During high-volume periods (August back-to-school), your payouts can exceed incoming payments on a given day. Model this in cash flow planning.
  • [✓]
    Confirm delivery race condition — idempotent handler. A buyer might double-click "Confirm Delivery". The escrow release handler must check escrow.status === 'HOLDING' before proceeding. Use a Prisma transaction with a where: { status: 'HOLDING' } guard so concurrent calls are safe.
Principal Engineer's Final Note

The three things that will cause production incidents if you skip them: (1) the webhook raw-body ordering — your Paystack signature verification will silently fail and every webhook will be rejected as unauthorized; (2) the idempotency check on webhooks — Paystack retries are not rare, they happen on any slow response, and a double-processed charge.success event will double-credit a seller's wallet; (3) the Decimal type on all money fields — floating-point rounding errors in a financial ledger accumulate and are nearly impossible to debug after the fact. Get these three right before writing any other code and the rest is recoverable.
