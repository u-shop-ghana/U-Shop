Section 01

Global Communication Standards

Every HTTP exchange between the Next.js client and the Express API must conform to the conventions defined in this section. No endpoint is exempt. These standards eliminate ambiguity and allow engineers to build on either side of the boundary without coordination.

1.1 Base URL Structure

All Express API routes are namespaced under /api/v1. This versioning prefix allows a future /api/v2 to co-exist without breaking existing clients during migrations.

Environment URLs
// Development
const API_BASE = "http://localhost:4000/api/v1"

// Production (Express on Fly.io / Railway)
const API_BASE = "https://api.ushop.com/api/v1"

// Store in Next.js environment variable — NEVER hardcoded in components
// .env.local
NEXT_PUBLIC_API_URL=http://localhost:4000/api/v1   // dev
NEXT_PUBLIC_API_URL=https://api.ushop.com/api/v1  // prod

The Paystack webhook endpoint is a special-case exception — it lives at /webhooks/paystack (no /api/v1 prefix) because it is not a client-facing endpoint and must be registered before Express's JSON body parser middleware. See Section 3 for full details.

1.2 Authentication Flow

U-Shop uses Supabase Auth as the identity provider. The JWT it issues is the single credential the Express backend trusts. The flow is strictly one-directional — Next.js reads the session from Supabase and forwards the token to Express; Express verifies it independently using the Supabase service role SDK.

1

User signs in via Supabase Auth (Next.js client)

Supabase issues a signed JWT access token and stores it in a HttpOnly cookie (sb-[project]-auth-token). The Next.js createBrowserClient handles this automatically.

2

Next.js reads the session token

In Server Components / Server Actions, use createServerClient from @supabase/ssr. In Client Components, use supabase.auth.getSession(). Extract session.access_token.

3

Next.js attaches the token to every Express request

Pass the token in the Authorization header as a Bearer token. Also always include Content-Type: application/json.

4

Express verifies the JWT

The authenticate middleware on Express calls supabase.auth.getUser(token) using the service role Supabase client. On success it attaches req.user (the verified Supabase user object) for use by downstream route handlers.

Next.js — How to call the Express API (utility function)
// lib/api.ts — reusable fetch wrapper for all Express API calls
import { createServerClient } from '@supabase/ssr'
import { cookies } from 'next/headers'

const API_URL = process.env.NEXT_PUBLIC_API_URL

/** Server-side authenticated fetch (Server Components, Server Actions) */
export async function apiFetch<T>(
  path: string,
  options?: RequestInit
): Promise<T> {
  const cookieStore = await cookies()
  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    { cookies: { getAll: () => cookieStore.getAll() } }
  )

  const { data: { session } } = await supabase.auth.getSession()
  if (!session) throw new Error('Unauthenticated')

  const res = await fetch(`${API_URL}${path}`, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${session.access_token}`,
      ...(options?.headers ?? {}),
    },
  })

  if (!res.ok) {
    const err = await res.json()
    throw err  // Throws the standardized ApiError object — see Section 4
  }

  return res.json()
}
Express — authenticate middleware
// middleware/authenticate.ts
import { createClient } from '@supabase/supabase-js'
import type { Request, Response, NextFunction } from 'express'

// Use the SERVICE ROLE key — bypasses RLS, never expose to frontend
const supabaseAdmin = createClient(
  process.env.SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!,
)

export async function authenticate(
  req: Request, res: Response, next: NextFunction
) {
  const authHeader = req.headers.authorization
  if (!authHeader?.startsWith('Bearer ')) {
    return res.status(401).json(buildError(401, 'MISSING_TOKEN', 'Authorization header required'))
  }

  const token = authHeader.slice(7)
  const { data: { user }, error } = await supabaseAdmin.auth.getUser(token)

  if (error || !user) {
    return res.status(401).json(buildError(401, 'INVALID_TOKEN', 'Token is invalid or expired'))
  }

  // Attach verified user to request — available in all downstream handlers
  req.user = user
  next()
}

/** Extend this with requireSeller / requireAdmin guards */
export async function requireSeller(
  req: Request, res: Response, next: NextFunction
) {
  const dbUser = await prisma.user.findUnique({ where: { supabaseId: req.user!.id } })
  if (!dbUser || (dbUser.role !== 'SELLER' && dbUser.role !== 'BOTH')) {
    return res.status(403).json(buildError(403, 'FORBIDDEN', 'Seller role required'))
  }
  req.dbUser = dbUser
  next()
}

1.3 Standardized Response Envelope

Every response from the Express API — success or error — is wrapped in a consistent JSON envelope. This means the Next.js client never has to guess the shape of a response; it always reads the same top-level fields.

✅ Success Envelope
TypeScript Interface
interface ApiSuccess<T> {
  success: true
  data: T
  meta?: {
    page: number
    limit: number
    total: number
    totalPages: number
  }
}

// Example: GET /listings
{
  "success": true,
  "data": [ { ...listing }, ... ],
  "meta": {
    "page": 1,
    "limit": 20,
    "total": 143,
    "totalPages": 8
  }
}
❌ Error Envelope
TypeScript Interface
interface ApiError {
  success: false
  error: {
    code: string    // Machine-readable
    message: string // Human-readable
    details?: Record<string, string[]>
    // ^ Zod field errors: { "price": ["Too small"] }
  }
}

// Example: POST /listings (validation fail)
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Request body failed validation",
    "details": {
      "price": ["Must be greater than 0"],
      "title": ["Required"]
    }
  }
}
Express — buildError helper (use in every route handler)
// utils/response.ts
export function buildSuccess<T>(data: T, meta?: ApiSuccess<T>['meta']) {
  return { success: true, data, ...(meta ? { meta } : {}) }
}

export function buildError(
  status: number, code: string, message: string,
  details?: Record<string, string[]>
) {
  return { success: false, error: { code, message, ...(details ? { details } : {}) } }
}

// Usage in a route handler:
res.status(201).json(buildSuccess(newStore))
res.status(409).json(buildError(409, 'SLUG_TAKEN', 'This store URL is already taken'))

2.3 Vendor Onboarding — Payout Recipient

Before a seller can receive any payouts, they must register their MoMo number or bank account with Paystack via the Express backend. Express calls the Paystack Create Transfer Recipient API and stores the resulting recipient_code on the seller's record. This code is used to initiate all future transfers to this seller.

⚠️

Security: The Express backend must own this call entirely. The frontend never calls the Paystack API directly. The PAYSTACK_SECRET_KEY lives in Express's environment only. A buyer must not be able to call this endpoint — apply the requireSeller guard.

POST
/api/v1/payouts/recipients
🏪 Seller Only
Registers a seller's MoMo number or bank account as a Paystack Transfer Recipient. Express calls POST https://api.paystack.co/transferrecipient on the seller's behalf, then persists the recipient_code and recipientId to the database. Idempotent — updating account details replaces the existing recipient.

2.5 The Escrow Flow — Checkout & Release

This is the most security-critical flow in U-Shop. The platform collects 100% of the order value from the buyer via Paystack, holds it in escrow, and releases the seller's cut only after the buyer confirms delivery. Below is the complete sequence before we define each endpoint.

1

Buyer clicks "Buy Now" → Next.js calls POST /api/v1/orders

Express creates an Order (status: PENDING_PAYMENT), generates a unique Paystack reference, and initializes a Paystack transaction for the full listing price. Returns the Paystack authorization_url.

2

Next.js redirects buyer to Paystack's hosted checkout

Buyer pays via MoMo or card at authorization_url. Paystack collects 100% of the funds to your platform account.

3

Paystack fires charge.success webhook → Express POST /webhooks/paystack

Express verifies the HMAC signature, marks the order as PAYMENT_RECEIVED, creates the Escrow record (status: HOLDING), and deducts seller platform fee. Schedules 7-day auto-release job.

4

Buyer confirms delivery → Next.js calls POST /orders/:id/release-escrow

Express verifies the caller is the buyer of this order, checks escrow.status === 'HOLDING', fires the Paystack Transfers API for the seller's net amount, and moves the order to COMPLETED pending the transfer.success webhook.

5

Paystack fires transfer.success webhook

Express sets escrow.status = 'RELEASED', credits the seller's wallet balance, and marks the order as COMPLETED.

POST
/api/v1/orders
🔒 Authenticated (Buyer)
Initializes a Paystack checkout session. Express creates an Order record, calls POST https://api.paystack.co/transaction/initialize with the full listing price (in kobo), and returns the hosted checkout URL. A seller cannot buy their own listing — Express must validate listing.storeId !== req.dbUser.store.id.
🔴

Money precision: All amounts sent to Paystack must be in kobo (integer, GHS × 100). Example: GHS 450.00 = kobo 45000. Use Decimal arithmetic throughout — never JavaScript floats. Store prices in the DB as Decimal and only convert to kobo at the point of the Paystack API call.

POST
/api/v1/orders/:orderId/release-escrow
🔒 Authenticated (Buyer of this order) CRITICAL
The buyer explicitly confirms they have received the item, triggering the payout to the seller. This is the most sensitive endpoint in the system. Express must enforce multiple guards before calling the Paystack Transfers API.
🔴

Guards — all must pass in order:
1. order.buyerId === req.user.id — only the buyer of this specific order can release it.
2. order.status === 'PAYMENT_RECEIVED' || 'PROCESSING' || 'IN_TRANSIT' — payment must have cleared.
3. escrow.status === 'HOLDING' — must use a Prisma transaction with a where: { status: 'HOLDING' } clause to prevent race conditions on double-click.
4. Seller must have a registered Paystack recipientCode — if missing, return 400 NO_PAYOUT_ACCOUNT.

// services/escrow.ts
const PLATFORM_FEE_RATE = new Decimal('0.05')  // 5% — adjust per PRD

async function releaseEscrow(orderId: string, buyerId: string) {
  return prisma.$transaction(async (tx) => {
    // Guard: atomically claim the escrow before any Paystack call
    const escrow = await tx.escrow.update({
      where: { orderId, status: 'HOLDING' },   // Will throw if not HOLDING
      data: { status: 'RELEASING' },             // Intermediate state prevents retry
    })

    const gross = new Decimal(escrow.amount)
    const platformFee = gross.mul(PLATFORM_FEE_RATE).toDecimalPlaces(2)
    const sellerNet = gross.sub(platformFee)
    const koboAmount = sellerNet.mul(100).toInteger().toNumber()

    // Fire Paystack Transfer
    const transfer = await PaystackService.initiateTransfer({
      source: 'balance',
      amount: koboAmount,
      recipient: seller.recipientCode,
      reason: `U-Shop payout for order ${orderId}`,
    })

    // Persist transfer_code — needed to reconcile transfer.success webhook
    await tx.payout.create({
      data: { sellerId, orderId, amount: sellerNet, transferCode: transfer.transfer_code, status: 'PROCESSING' }
    })

    return { transferCode: transfer.transfer_code, sellerNet, platformFee }
  })
}

Section 03

Paystack Webhook Specifications

Paystack notifies your Express backend of payment and transfer events by sending an HTTP POST request to a registered URL. This section defines the exact endpoint, the full signature verification algorithm, and the expected handler logic for each event type.

🔴

Critical Middleware Ordering: The webhook route MUST be registered in index.ts BEFORE app.use(express.json()). The HMAC-SHA512 signature is computed over the raw request body bytes. Once express.json() parses the body, the raw buffer is gone and signature verification will silently fail on every single webhook. This is a production-breaking mistake.

index.ts — correct middleware order
import express from 'express'
import { webhookRouter } from './routes/webhooks'
import { apiRouter } from './routes/api'

const app = express()

// ── 1. WEBHOOK ROUTE MUST COME FIRST — before JSON parser ──────────────
// Uses express.raw() to preserve the raw body buffer for HMAC verification
app.use('/webhooks', express.raw({ type: 'application/json' }), webhookRouter)

// ── 2. JSON parser for all other routes ────────────────────────────────
app.use(express.json())
app.use(express.urlencoded({ extended: true }))

// ── 3. API routes ──────────────────────────────────────────────────────
app.use('/api/v1', apiRouter)

3.1 Endpoint & Signature Verification

POST /webhooks/paystack
The URL you register in your Paystack Dashboard → Settings → Webhooks. No authentication header is needed — the HMAC signature IS the authentication.
routes/webhooks.ts — full signature verification + idempotency guard
import { Router } from 'express'
import crypto from 'crypto'

const router = Router()
const PAYSTACK_SECRET = process.env.PAYSTACK_SECRET_KEY!

router.post('/', async (req, res) => {
  // ── Step 1: Verify HMAC-SHA512 signature ────────────────────────────
  // req.body is a Buffer here (express.raw middleware) — NOT a parsed object
  const signature = req.headers['x-paystack-signature'] as string
  const expectedSig = crypto
    .createHmac('sha512', PAYSTACK_SECRET)
    .update(req.body)         // Pass the raw Buffer — NOT JSON.stringify(body)
    .digest('hex')

  if (!crypto.timingSafeEqual(
    Buffer.from(signature, 'hex'),
    Buffer.from(expectedSig, 'hex')
  )) {
    return res.status(401).send('Invalid signature')
  }

  // ── Step 2: Parse body now that we've verified it ───────────────────
  const event = JSON.parse(req.body.toString()) as PaystackEvent

  // ── Step 3: Idempotency guard ────────────────────────────────────────
  // Paystack retries webhooks. The WebhookEvent table with a unique
  // constraint on externalId prevents double-processing.
  const externalId = event.data.id.toString()  // Paystack's transaction/transfer ID
  const existing = await prisma.webhookEvent.findUnique({ where: { externalId } })
  if (existing) {
    return res.status(200).send('Already processed')  // Acknowledge to stop retries
  }

  // ── Step 4: Persist event record (dedupe lock) ───────────────────────
  await prisma.webhookEvent.create({
    data: { externalId, event: event.event, payload: event as any }
  })

  // ── Step 5: Respond 200 IMMEDIATELY — before heavy processing ───────
  // Paystack considers any non-2xx response a failure and will retry.
  // Acknowledge receipt first, then do the work asynchronously.
  res.status(200).send('OK')

  // ── Step 6: Route to correct handler ────────────────────────────────
  switch (event.event) {
    case 'charge.success':
      await handleChargeSuccess(event.data)
      break
    case 'transfer.success':
      await handleTransferSuccess(event.data)
      break
    case 'transfer.failed':
      await handleTransferFailed(event.data)
      break
    case 'transfer.reversed':
      await handleTransferReversed(event.data)
      break
    default:
      console.log(`Unhandled Paystack event: ${event.event}`)
  }
})

export const webhookRouter = router

3.2 Event: charge.success

Fired when a buyer's payment is confirmed. This is the trigger that moves the order from PENDING_PAYMENT to PAYMENT_RECEIVED and creates the escrow record.

Paystack Payload Shape
interface ChargeSuccessData {
  id: number                // Paystack transaction ID — use as externalId
  reference: string        // Matches paystackRef on the Order record
  amount: number           // In kobo — divide by 100 for GHS
  currency: string         // "GHS"
  status: string           // "success"
  paid_at: string          // ISO 8601
  channel: string          // "mobile_money" | "card"
  customer: {
    email: string
    phone: string | null
  }
  metadata: {
    orderId: string        // ⬅ You embed this when initializing the transaction
    listingId: string
    buyerId: string
  }
}
handleChargeSuccess — database operations
async function handleChargeSuccess(data: ChargeSuccessData) {
  const { orderId, listingId } = data.metadata
  const amountGHS = new Decimal(data.amount).div(100)

  await prisma.$transaction(async (tx) => {
    // 1. Update order status
    await tx.order.update({
      where: { id: orderId },
      data: { status: 'PAYMENT_RECEIVED', paidAt: new Date(data.paid_at) }
    })

    // 2. Create escrow record — funds are now "held" on the platform
    await tx.escrow.create({
      data: {
        orderId,
        amount: amountGHS,
        status: 'HOLDING',
        autoReleaseAt: new Date(Date.now() + 7 * 24 * 60 * 60 * 1000),
      }
    })

    // 3. Mark listing as SOLD to prevent double-purchase
    await tx.listing.update({
      where: { id: listingId },
      data: { status: 'SOLD' }
    })
  })

  // 4. Schedule auto-release job outside the transaction
  await scheduleEscrowReminders(orderId, autoReleaseAt)
  await notifySellerOfNewOrder(orderId)
}

3.3 Event: transfer.success

Fired when the Paystack transfer to the seller's MoMo or bank account is confirmed. This finalizes the escrow release and credits the seller's in-platform wallet.

Paystack Payload Shape + handleTransferSuccess
interface TransferData {
  id: number
  transfer_code: string   // Matches Payout.transferCode in DB
  amount: number          // Kobo
  status: string          // "success" | "failed" | "reversed"
  recipient: {
    recipient_code: string
    name: string
    details: { account_number: string; bank_code: string }
  }
}

async function handleTransferSuccess(data: TransferData) {
  // Look up the Payout record using transfer_code — NOT order ID
  const payout = await prisma.payout.findUniqueOrThrow({
    where: { transferCode: data.transfer_code },
    include: { order: { select: { id: true, sellerId: true } } }
  })

  await prisma.$transaction(async (tx) => {
    // 1. Finalize escrow
    await tx.escrow.update({
      where: { orderId: payout.orderId },
      data: { status: 'RELEASED', releasedAt: new Date() }
    })

    // 2. Mark payout complete
    await tx.payout.update({
      where: { id: payout.id },
      data: { status: 'COMPLETED', completedAt: new Date() }
    })

    // 3. Mark order as COMPLETED
    await tx.order.update({
      where: { id: payout.orderId },
      data: { status: 'COMPLETED' }
    })

    // 4. Credit seller's in-platform wallet (for future reference / withdrawals)
    const amountGHS = new Decimal(data.amount).div(100)
    await tx.wallet.update({
      where: { userId: payout.order.sellerId },
      data: { balance: { increment: amountGHS } }
    })
  })

  await notifySellerOfPayout(payout.orderId, amountGHS)
}

3.4 Event: transfer.failed

Fired when a Paystack transfer to a seller is rejected (e.g., invalid MoMo number, insufficient Paystack balance). The funds remain on your platform and the escrow should be reverted to HOLDING so it can be retried.

handleTransferFailed
async function handleTransferFailed(data: TransferData) {
  const payout = await prisma.payout.findUniqueOrThrow({
    where: { transferCode: data.transfer_code }
  })

  await prisma.$transaction(async (tx) => {
    // Revert escrow to HOLDING so admin/seller can retry
    await tx.escrow.update({
      where: { orderId: payout.orderId },
      data: { status: 'HOLDING' }
    })
    await tx.payout.update({
      where: { id: payout.id },
      data: { status: 'FAILED', failedAt: new Date(), failureReason: data.status }
    })
  })

  // Alert: send internal Slack/email alert for manual review
  await alertAdminOfFailedTransfer(payout, data)
}
Idempotency & Reconciliation

Paystack retries webhooks up to 3 times on non-2xx responses or timeouts. The WebhookEvent table with a UNIQUE constraint on externalId (Paystack's data.id) is your primary defense. Additionally, run a cron job every 5 minutes to query Paystack's GET /transaction/verify/:reference for any orders that have been in PENDING_PAYMENT for more than 10 minutes. This reconciles orders where your server was down during webhook delivery.

Section 04

Standardized Error Handling

Every error response from the Express API must follow the same ApiError envelope defined in Section 1.3. Below is the complete mapping of HTTP status codes used in this API and their corresponding machine-readable error codes.

4.1 HTTP Status Code Map

400
Bad Request
The request is structurally malformed (not JSON, missing required field that Zod can't parse). Distinct from 422. Common codes: MALFORMED_JSON, MISSING_BODY.
401
Unauthorized
No token provided, token is expired, or token signature is invalid. The client should redirect to login. Common codes: MISSING_TOKEN, INVALID_TOKEN, TOKEN_EXPIRED.
403
Forbidden
Token is valid but the user lacks the permission for this action (e.g., a buyer calling a seller-only endpoint, a user accessing another user's order). Common codes: FORBIDDEN, SELLER_ROLE_REQUIRED, NOT_YOUR_ORDER.
404
Not Found
The requested resource does not exist or is intentionally hidden from the caller. Never use 200 with an empty body for not-found. Common codes: LISTING_NOT_FOUND, ORDER_NOT_FOUND, STORE_NOT_FOUND.
409
Conflict
The request conflicts with the current state of a resource. The primary use case is slug uniqueness collisions and duplicate resource creation. Common codes: SLUG_TAKEN, STORE_EXISTS, DUPLICATE_WEBHOOK.
422
Unprocessable Entity
The JSON is valid and parseable, but the field values fail business-rule validation (Zod schema). The error.details map shows field-by-field errors. Common codes: VALIDATION_ERROR. The details field is always populated for 422.
500
Internal Server Error
An unexpected server-side error. In production, message is always a generic string — never expose stack traces or DB errors to clients. Log the full error to Sentry. Common code: INTERNAL_ERROR.

4.3 Global Error Handler (Express)

Register this as the final middleware in index.ts. It catches any unhandled errors, logs them to Sentry, and ensures they are always returned in the standardized envelope.

middleware/errorHandler.ts
import * as Sentry from '@sentry/node'
import type { Request, Response, NextFunction } from 'express'
import { ZodError } from 'zod'
import { Prisma } from '@prisma/client'

export function globalErrorHandler(
  err: unknown, req: Request, res: Response, _next: NextFunction
) {
  // ── ZodError: validation failure ───────────────────────────────────────
  if (err instanceof ZodError) {
    const details: Record<string, string[]> = {}
    for (const issue of err.issues) {
      const field = issue.path.join('.') || '_root'
      details[field] = [...(details[field] ?? []), issue.message]
    }
    return res.status(422).json(buildError(422, 'VALIDATION_ERROR',
      'Request body failed validation', details))
  }

  // ── Prisma NOT FOUND: P2025 ────────────────────────────────────────────
  if (err instanceof Prisma.PrismaClientKnownRequestError
      && err.code === 'P2025') {
    return res.status(404).json(buildError(404, 'NOT_FOUND', 'Resource not found'))
  }

  // ── Prisma UNIQUE VIOLATION: P2002 ────────────────────────────────────
  if (err instanceof Prisma.PrismaClientKnownRequestError
      && err.code === 'P2002') {
    return res.status(409).json(buildError(409, 'CONFLICT', 'A resource with this value already exists'))
  }

  // ── Unknown error: report to Sentry, return generic 500 ───────────────
  Sentry.captureException(err)
  // Log full error server-side only — never expose to client
  console.error('[UNHANDLED ERROR]', err)
  return res.status(500).json(buildError(500, 'INTERNAL_ERROR',
    'An unexpected error occurred. Our team has been notified.'))
}

// In index.ts — must be registered LAST, after all routes
app.use(globalErrorHandler)

How the frontend reads error codes
// In a Next.js Server Action or Client Component
try {
  const { data } = await apiFetch('/stores', { method: 'POST', body: ... })
} catch (err) {

    case 'SLUG_TAKEN':

      router.push('/login?reason=session-expired')
      break
    default:
      toast.error(apiError.error?.message ?? 'Something went wrong')
  }
}
Principal Engineer's Contract Summary

Three things will cause a production incident if you skip them: (1) Webhook raw-body ordering — registering the Paystack webhook route after express.json() means every single webhook is silently rejected with a signature mismatch. (2) Idempotency on webhooks — Paystack retries on any slow response; without the WebhookEvent deduplication table, a retry double-credits a wallet or double-releases escrow. (3) Decimal type for all money — storing prices or escrow amounts as JavaScript floats introduces rounding errors that accumulate silently in a financial ledger. Every price, fee, and wallet balance must use Prisma's Decimal field and be converted to kobo (integer) only at the moment of the Paystack API call.
